# CharPick 逐章提取与Summary提取设计详解

## 概述

CharPick 的人设提取系统采用**两层分层架构**：
- **L2 层（逐章提取层）**：按章节进行细粒度提取，捕捉每章的角色、剧情、物品、地点信息
- **L3 层（汇总层）**：基于全书所有章节的提取结果进行全局汇总，生成角色总表、物品总表、时间线等

---

## 第一部分：逐章提取设计（L2 - Chapter Extraction）

### 1.1 基本概念

逐章提取是对**单个章节**进行结构化信息提取，目标是从一章文本中识别出以下维度的信息：
- 章节信息（Chapter Info）
- 剧情（Plot）
- 角色（Characters）
- 物品（Items）
- 地点与世界观（World）

### 1.2 提取的五个维度

#### 维度 1：章节信息（chapter_info）
**提取指令**：提取章节信息：章节号、章节名、故事时间信息、出场角色。若无相关信息则不要臆造。

**输出结构**：
```json
{
  "chapter_number": "1",
  "chapter_title": "序章",
  "time_info": "某年某月某日，午时",
  "appearing_characters": ["石野", "小王"]
}
```

**设计要点**：
- 章节号和章节名从文本标题直接识别
- 故事时间信息涉及小说的世界观时间线
- 出场角色仅列出在本章出现过的人物

---

#### 维度 2：剧情（plot）
**提取指令**：提取本章剧情：章节概要、关键事件。若无关键事件则不输出该字段。

**输出结构**：
```json
{
  "summary": "石野第一次来到城市，发现了一个神秘的地下组织。",
  "key_events": [
    "石野踏入城市大门",
    "遇见了黑衣人的追踪",
    "发现了地下室的秘密"
  ]
}
```

**设计要点**：
- 章节概要：用一句话概括本章的核心情节
- 关键事件：提取 2-5 个影响剧情发展的重要事件
- 如果章节内没有明显的关键事件，可省略该字段

---

#### 维度 3：角色（characters）
**提取指令**：提取本章角色：姓名、性别、年龄、种族、社会关系、外表描述、本章经历行为、性格表现、人物台词。若无则不输出。

**输出结构**：
```json
{
  "characters": [
    {
      "name": "石野",
      "gender": "男",
      "age": "23",
      "race": "人类",
      "relationships": [
        {
          "target": "小王",
          "type": "陌生人（后转为同盟者）"
        }
      ],
      "appearance": "高个子，棕色长发，左眼有一道伤疤",
      "chapter_behavior": "谨慎地观察周围环境，多次询问关于城市的信息",
      "personality": "警惕、聪慧、冷静",
      "dialogue": [
        "这座城市看起来不安全。",
        "你是谁？为什么跟踪我？"
      ]
    }
  ]
}
```

**设计要点**：
- 每个角色一个对象，存储在数组中
- 社会关系特指**本章出现的**角色之间的关系
- 外表描述：仅从文本中提取，不进行臆造
- 本章经历行为：描述角色在这一章做了什么
- 性格表现：提取本章体现出的性格特征（可简化为几个形容词）
- 人物台词：提取 2-3 句代表性的台词或对话

---

#### 维度 4：物品（items）
**提取指令**：提取本章关键物品：物品名、描述、本章表现、所属角色。若无则不输出。

**输出结构**：
```json
{
  "items": [
    {
      "name": "黑色怀表",
      "description": "古董怀表，表盘上有神秘符号",
      "chapter_appearance": "石野从地下室发现的神秘物品，后被小王识别为组织的标志",
      "owner": "不详（可能属于地下组织）"
    }
  ]
}
```

**设计要点**：
- 仅提取对剧情有影响的"关键物品"，日常用品可忽略
- 物品描述应来自文本描写
- 本章表现：描述该物品在本章的作用和变化（是否被获得、转移、销毁等）
- 所属角色：明确物品的持有者或原有者

---

#### 维度 5：地点与世界观（world）
**提取指令**：提取本章地点与背景设定：出现地点、本章提及世界观。若无则不输出。

**输出结构**：
```json
{
  "locations": [
    {
      "name": "中央城市",
      "description": "一座高科技与魔法融合的城市，摩天楼与古老遗迹并存",
      "features": "繁忙的商业街、神秘的地下空间、古老的神庙遗迹"
    }
  ],
  "worldview": {
    "setting": "科幻魔法融合世界",
    "factions": ["地下组织", "官方势力"],
    "key_concepts": ["时空裂隙", "神秘能量"]
  }
}
```

**设计要点**：
- 地点：提取本章明确出现过的位置
- 世界观：包括设定、势力、关键概念等宏观信息
- 如果本章未涉及新的地点或世界观信息，可省略

---

### 1.3 逐章提取的技术实现

#### 核心流程（来自 `chapter_structured_extractor.py`）

```python
def extract_chapter_structured(
    chapter_title: str,
    chapter_text: str,
    enabled_dimensions: list[str] | None = None,
) -> dict:
    """
    对单个章节进行结构化提取
    
    参数：
    - chapter_title: 章节标题
    - chapter_text: 章节内容（限制为前 6000 字）
    - enabled_dimensions: 启用的提取维度列表（默认为全部）
    
    返回：提取结果的字典
    """
```

#### 实现特点

1. **多维度支持**：支持灵活选择要提取的维度
2. **长文本截断**：处理前 6000 字以防止 LLM 超时
3. **空字段清理**：自动移除空值和空对象，保持输出简洁
4. **错误捕获**：如某维度提取失败，记录错误但继续处理其他维度
5. **LLM 调用**：通过 `call_ollama_json()` 函数调用本地/远程 LLM

#### 配置示例（config.json）

```json
{
  "chapter_prompts": {
    "chapter_info": "提取章节信息：章节号、章节名、故事时间信息、出场角色。若无相关信息则不要臆造。",
    "plot": "提取本章剧情：章节概要、关键事件。若无关键事件则不输出该字段。",
    "characters": "提取本章角色：姓名、性别、年龄、种族、社会关系、外表描述、本章经历行为、性格表现、人物台词。若无则不输出。",
    "items": "提取本章关键物品：物品名、描述、本章表现、所属角色。若无则不输出。",
    "world": "提取本章地点与背景设定：出现地点、本章提及世界观。若无则不输出。"
  }
}
```

## 第二部分：Summary 提取设计（L3 - Global Summarization）

### 2.1 基本概念

Summary 提取是对**整本书**的所有章节提取结果进行**全局汇总和聚合**，目的是：
- 消除重复和冲突的信息
- 合并分散在各章节中的角色信息
- 构建完整的角色、物品、剧情、地点的全局画像
- 为后续的角色卡生成（L4）提供统一的数据源

### 2.2 Summary 的四个维度

#### 维度 1：角色总表（characters）
**提取指令**：基于整本书的全部章节提取角色总表。请严格输出 JSON 数组，不要输出 Markdown 代码块，不要输出额外解释。数组中的每个元素都是一个对象，且至少包含 name、summary、identity、appearance、personality 这五个字段；如信息不足，字段可留空字符串，但不要编造。

**输出结构**：
```json
[
  {
    "name": "石野",
    "summary": "主角，一个神秘的流浪者。在踏入城市后发现了地下组织的秘密，并逐渐成长为反抗势力的领导者。",
    "identity": "流浪者/反抗组织首领",
    "appearance": "高个子，棕色长发，左眼有一道伤疤，气质冷漠但眼神深邃",
    "personality": "警惕、聪慧、冷静、富有同情心、隐藏着过去的创伤"
  },
  {
    "name": "小王",
    "summary": "第二章出现的城市本地人，最初与石野是敌对关系，后成为盟友。",
    "identity": "城市导游/情报员",
    "appearance": "矮个子，黑色短发，穿着整洁的工作服",
    "personality": "热心、机灵、善于社交"
  }
]
```

**设计要点**：
- **name**：角色的正式名字或常用昵称
- **summary**：角色在整本书中的角色定位和发展轨迹（2-3句）
- **identity**：角色的身份或社会角色
- **appearance**：从全书综合得出的完整外表描述
- **personality**：汇总全书中体现的性格特征

---

#### 维度 2：物品总表（items）
**提取指令**：基于整本书的全部章节提取物品总表。请严格输出 JSON 数组，不要输出 Markdown 代码块，不要输出额外解释。数组中的每个元素都是一个对象，且至少包含 name、summary 两个字段；必要时可补充 identity、appearance、personality 等字段，但不要编造。

**输出结构**：
```json
[
  {
    "name": "黑色怀表",
    "summary": "神秘的古董怀表，表盘上刻有未知的符号。在第一章被石野发现，后被识别为地下组织的通行证。在第五章因被破坏而失效，触发了关键的剧情转折。"
  },
  {
    "name": "蓝色手册",
    "summary": "组织的内部秘密文件，包含组织的历史和成员信息。石野在第三章获得，通过它了解了更多关于地下世界的真相。"
  }
]
```

**设计要点**：
- **name**：物品的名称
- **summary**：物品的完整介绍，包括其作用、在故事中的地位、被转移的轨迹等
- 可选字段：如物品有特殊的"性格"或"特征"，可补充 appearance 等字段

---

#### 维度 3：剧情时间线（storyline_events）
**提取指令**：基于整本书的全部章节提取剧情时间线。请输出便于用户阅读的 Markdown 格式，不要输出 JSON，不要输出代码块包裹。建议使用标题、编号列表或表格呈现章节顺序、关键事件、因果关系、时间推进与状态变化；若信息不足，请直接省略，不要编造。

**输出示例**：
```markdown
# 剧情时间线

## 序章（第 1 章）
- 石野踏入中央城市
- 遇见了黑衣人的追踪
- 发现了地下室的秘密

## 第二章：城市初体验
- 石野认识了小王
- 初步了解城市的社会结构
- 被通缉令追捕

## 第三章：真相渐露
- 石野获得了蓝色手册
- 了解到地下组织的历史
- 决定加入反抗势力

...

## 关键转折
| 事件 | 章节 | 影响 |
|------|------|------|
| 怀表破坏 | 第 5 章 | 地下组织暴露 |
| 同盟形成 | 第 8 章 | 反抗力量壮大 |
```

**设计要点**：
- 按章节顺序组织信息
- 重点突出关键事件和转折点
- 可用表格、列表、标题等 Markdown 元素增强可读性
- **重点**：这个维度**不输出 JSON，而是易读的 Markdown**

---

#### 维度 4：世界观与地点（world_locations）
**提取指令**：基于整本书的全部章节提取世界观与地点信息。请输出便于用户阅读的 Markdown 格式，不要输出 JSON，不要输出代码块包裹。建议使用标题、分级小节、列表或表格呈现地点、场景、势力分布、地域关系与世界观设定；若信息不足，请直接省略，不要编造。

**输出示例**：
```markdown
# 世界观与地点信息

## 地理设定
### 中央城市
- **位置**：大陆中心
- **特征**：高科技与魔法融合
- **主要区域**：
  - 商业街（繁华、高度控制）
  - 地下空间（隐秘、黑市汇集地）
  - 古老神庙遗迹（神秘、力量源地）

### 郊外异族聚集地
- **位置**：城市外围
- **特征**：自治、多元文化融合
- **主要势力**：异族联盟

## 势力分布
| 势力 | 属性 | 势力范围 |
|------|------|--------|
| 官方政权 | 中央集权 | 城市主要区域 |
| 地下组织 | 反抗势力 | 地下空间、郊外 |
| 异族联盟 | 独立 | 郊外聚集地 |

## 核心世界观设定
- **时空裂隙**：导致两个世界产生交融
- **神秘能量**：源自远古遗迹，驱动整个世界
- **种族平衡**：人类与异族的共存矛盾
```

**设计要点**：
- 用分级标题组织地理信息
- 用表格展现势力和关系
- **重点**：这个维度**不输出 JSON，而是易读的 Markdown**

### 2.3 Summary 提取的技术实现

#### 工作流程（来自 `llm_dispatcher.py`）

Summary 提取通常在逐章提取完成后执行：

1. **收集所有章节的 L2 提取结果**
2. **构建聚合提示词**：将全部章节信息作为上下文传给 LLM
3. **调用 LLM 进行汇总**：LLM 根据提示词的指导进行聚合和总结
4. **处理输出格式**：
   - JSON 维度（characters、items）：解析并验证
   - Markdown 维度（storyline_events、world_locations）：保留原始格式
5. **存储结果**：保存到 L3 层（summary 表）

#### 配置示例（config.json）

```json
{
  "summary_prompts": {
    "characters": "基于整本书的全部章节提取角色总表。请严格输出 JSON 数组...",
    "items": "基于整本书的全部章节提取物品总表。请严格输出 JSON 数组...",
    "storyline_events": "基于整本书的全部章节提取剧情时间线。请输出便于用户阅读的 Markdown 格式...",
    "world_locations": "基于整本书的全部章节提取世界观与地点信息。请输出便于用户阅读的 Markdown 格式..."
  }
}
```

## 第三部分：两层架构的协作

### 3.1 数据流转

```
原始文本
  ↓
文本预处理（章节分割、清洗）
  ↓
逐章提取（L2）← 调用 extract_chapter_structured()
  ├─ chapter_info
  ├─ plot
  ├─ characters
  ├─ items
  └─ world
  ↓
本地缓存和远程同步（保存到 local_cache 和 supabase）
  ↓
Summary 提取（L3）← 调用 LLM 进行全局聚合
  ├─ characters（总表）
  ├─ items（总表）
  ├─ storyline_events（时间线）
  └─ world_locations（世界观）
  ↓
存储到 summary 表
  ↓
角色卡生成（L4）← 基于 summary 中选中的角色
  ↓
前端展示和 LLM 交互
```

### 3.2 关键特点

| 特性 | L2 逐章提取 | L3 Summary 提取 |
|------|----------|----------------|
| **粒度** | 单章节 | 全书 |
| **时间复杂度** | 线性（逐章处理） | 低（全部章节作为上下文） |
| **输出格式** | JSON（结构化） | JSON + Markdown（混合） |
| **更新策略** | 增量更新（新增章节） | 重新计算（基于最新的所有章节） |
| **数据用途** | 详细查询、章节级调查 | 全局展示、角色总表、列表生成 |
| **容错能力** | 单章失败不影响其他 | 需要完整的章节输入 |

---

### 3.3 冗余设计的意义

为什么需要两层？

1. **查询灵活性**
   - L2：支持"第 3 章中石野的性格如何？"这样的**细粒度查询**
   - L3：支持"整本书中石野的完整人设是怎样的？"这样的**全局查询**

2. **性能优化**
   - L2：增量更新，新增章节只需重新提取那一章
   - L3：可以缓存并在用户需要时重新计算

3. **数据完整性**
   - L2：保留原始提取证据和章节上下文
   - L3：对信息进行去重、合并、归纳，形成统一的数据视图

4. **错误恢复**
   - L2 某章提取失败不影响其他章节
   - L3 可以手动调整或重新触发聚合

## 第五部分：设计考量与未来优化

### 5.1 当前设计的优势

1. ✅ **信息完整性**：两层架构确保了从微观到宏观的信息覆盖
2. ✅ **LLM 灵活性**：Prompt-based 设计，易于调整提取维度和质量
3. ✅ **容错机制**：单维度或单章节失败不影响整体
4. ✅ **可读性混合**：JSON 和 Markdown 的组合满足不同场景需求

### 5.2 已知的挑战与改进方向

1. **长文本漂移**（Character Drift in Long Texts）
   - 问题：角色性格在书的后期可能发生变化，Summary 可能偏向后期特征
   - 解决方向：采用 **Recursive Summarization** 或 **RAG 检索增强生成**，将章节按时间序列分组汇总

2. **关系复杂度**（Relationship Complexity）
   - 问题：角色之间的关系在全书中可能有多次变化，L3 难以完整表达
   - 解决方向：在 Summary 中添加 `relationship_evolution` 字段，记录关系的时间序列变化

3. **信息冗余与矛盾**（Data Redundancy and Conflicts）
   - 问题：不同章节中可能对同一角色有不同的描述（如年龄、外表变化）
   - 解决方向：实现基于时间线的属性变化追踪

4. **多模态集成**（Multimodal Integration）
   - 问题：目前只支持文本提取，无法识别图片中的角色或场景
   - 解决方向：集成 Vision LLM（如 GPT-4V）进行图文混合提取

---

## 总结

CharPick 的两层分层提取设计体现了**信息论和系统论**的思想：

- **L2（逐章提取）**代表 **微观层面**的高保真采集
- **L3（Summary提取）**代表 **宏观层面**的信息融合

通过这个设计，系统既能支持**细粒度的证据查询**，又能生成**全局的人设定位**，为后续的角色卡生成、LLM 交互等应用奠定了坚实的基础。
