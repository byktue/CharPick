# CharPick 角色卡（Character Card）设计详解

## 概述

角色卡是 CharPick 系统中的 **L4 层**（展示调度层），是向前端和用户呈现的**最终业务产品**。角色卡基于 Summary 提取结果（L3）和逐章提取结果（L2），通过一个精心设计的 Prompt 进行结构化生成，目的是创建一个**可立即调用的、功能完整的角色人设**。

---

## 第一部分：角色卡的生成流程

### 1.1 三层数据融合

```
L2 逐章提取结果
  ├─ 第1章的石野：外表描述 + 性格表现 + 对话
  ├─ 第3章的石野：行为 + 性格变化
  └─ 第8章的石野：领导力 + 新对话
         ↓
    (多章聚合)
         ↓
L3 Summary 提取结果
  └─ 石野（角色总表条目）
     ├─ name: "石野"
     ├─ summary: "主角，流浪者..."
     ├─ identity: "流浪者/反抗组织首领"
     ├─ appearance: "高个子，棕色长发..."
     └─ personality: "警惕、聪慧、冷静..."
         ↓
    (用户选择某个角色)
         ↓
L4 角色卡生成
  └─ 调用 Card Prompt，融合上述两层数据
     ↓
输出完整的角色卡
```

### 1.2 生成的核心步骤

#### 步骤 1：用户选择角色
```
前端展示：从 L3 summary.characters 的角色总表中
用户点击：选择感兴趣的角色（如"石野"）
```

#### 步骤 2：准备上下文数据
```python
# 系统内部操作
summary_character = summary.characters[selected_index]  # L3 数据
chapter_evidences = get_all_chapter_extractions_for_character(character_name)  # L2 数据
```

#### 步骤 3：调用 Card Prompt
```python
prompt = config["card_prompts"]["character_card"]
result = call_llm(
    prompt=prompt,
    context={
        "summary_character": summary_character,  # L3 全局画像
        "chapter_evidences": chapter_evidences,  # L2 详细证据
    }
)
```

#### 步骤 4：结构化解析与展示
```
LLM 输出的原始结构化结果（字典/JSON）
         ↓
解析并映射到标准字段
         ↓
生成前端展示卡片
         ↓
用户调用 LLM API 进行交互
```

## 第二部分：Card Prompt 的设计

### 2.1 当前的 Card Prompt（config.json）

```json
{
  "card_prompts": {
    "character_card": "基于 summary.characters 中选中的角色条目，并结合对应章节证据，生成可调用的角色展示卡：输出角色名称、简介、外形特征、性格标签、关键经历、经典台词、与其他角色关系，以及适合前端卡片展示的精简文案。若 summary 角色条目信息与章节证据不一致，请优先以章节证据修正，但不要臆造。"
  }
}
```

### 2.2 Prompt 的核心指导逻辑

#### 数据来源优先级
```
优先级 1 ： 章节证据（L2）
  ↓
优先级 2 ： Summary 条目（L3）
  ↓
优先级 3 ： 推理填补

原因：章节中的直接引文比汇总更准确，可以避免信息在汇总阶段的"漂移"
```

#### 质量约束
- ✅ 输出角色名称（确保身份）
- ✅ 简介（2-3 句话，整体定位）
- ✅ 外形特征（外观、气质）
- ✅ 性格标签（几个关键词，易于理解）
- ✅ 关键经历（全书中的转折点）
- ✅ 经典台词（代表性的对话）
- ✅ 与其他角色关系（关系网络）
- ✅ 前端文案（精简、吸引人的介绍）
- ❌ 不要臆造

## 第三部分：角色卡包含的要素

### 3.1 标准的 7 个核心字段

根据 `llm_dispatcher.py` 中的 `_card_markdown_from_result()` 函数，系统定义了**7 个标准展示字段**：

| # | 字段名 | 对应的 JSON Key（优先级顺序） | 含义 | 示例 |
|---|--------|-------------------------------|------|------|
| 1 | **简介** | `intro` / `summary` / `description` | 角色的整体定位和设定 | "石野是故事的主角，一位神秘的流浪者。在踏入城市后..." |
| 2 | **外形特征** | `appearance` / `look` | 外观、气质、标志性特征 | "高个子，棕色长发，左眼有一道伤疤，气质冷漠" |
| 3 | **性格标签** | `personality` / `traits` / `tags` | 性格概括，用 3-5 个词 | "警惕、聪慧、冷静、富有同情心、隐藏创伤" |
| 4 | **关键经历** | `key_events` / `experiences` / `events` | 全书中影响角色的关键转折点 | "踏入城市、获得蓝色手册、领导反抗势力..." |
| 5 | **常用台词** | `quotes` / `speech` / `sayings` | 代表性的对话或口头禅 | "这座城市看起来不安全。" / "跟我一起，我们会改变这个世界。" |
| 6 | **与其他角色关系** | `relations` / `relationships` / `relation` | 关系网络和互动 | "小王：盟友、导师\n官方首领：对立关系" |
| 7 | **前端展示文案** | `display_text` / `teaser` / `card_text` / `intro_text` | 精简的、吸引用户的文案 | "一个被迫成为领导者的流浪者..." |

### 3.2 字段映射与获取逻辑

系统采用**多键映射**机制，以提高 LLM 输出的容错性：

```python
field_map = [
    ("简介", ["intro", "summary", "description"]),
    ("外形特征", ["appearance", "look"]),
    ("性格标签", ["personality", "traits", "tags"]),
    ("关键经历", ["key_events", "experiences", "events"]),
    ("经典台词", ["quotes", "speech", "sayings"]),
    ("与其他角色关系", ["relations", "relationships", "relation"]),
    ("前端展示文案", ["display_text", "teaser", "card_text", "intro_text"]),
]
```

**获取逻辑**（fallback 机制）：
```
for each field (如"简介"):
    for each key (如["intro", "summary", "description"]):
        if key exists in LLM output:
            return that value  ← 找到就用
    
    # 如果 LLM 输出中没有，则从 Summary 中查找
    if not found in LLM output:
        for each key in field_map:
            if key exists in summary_character:
                return that value
    
    # 如果都找不到
    if still not found:
        return "-"  ← 用占位符
```

## 第五部分：角色卡的应用场景

### 5.1 前端展示

角色卡可以渲染成：
- **卡片形式**：头像 + 简介 + 标签
- **详情页面**：完整的角色档案
- **角色库列表**：多个角色的缩略卡片

### 5.2 LLM 调用

用户可以复制角色卡的内容，调用 LLM API 进行交互：

```python
system_prompt = f"""
你是一个角色扮演 AI。
【角色信息】
名字: {card['name']}
简介: {card['intro']}
外形: {card['appearance']}
性格: {', '.join(card['personality'])}
经历: {'; '.join(card['key_events'])}
台词: {card['quotes']}
关系: {card['relations']}

【要求】
1. 按照角色的性格和经历进行回复
2. 融合引用常用台词
3. 符合角色的设定
"""

response = llm_api.chat(
    messages=[{"role": "system", "content": system_prompt},
              {"role": "user", "content": user_input}]
)
```

### 5.3 本地记忆缓存

系统支持保存角色卡和对话历史，实现**本地对话记忆**：
```
用户与 石野 的对话记录：
- 用户输入
- AI 回复（基于角色卡生成）
- 记忆缓存
- 下一轮对话继续引用记忆
```

## 第六部分：角色卡设计的关键决策

### 6.1 为什么需要两个信息源（L3 + L2）？

| 方面 | L3 Summary | L2 逐章 | 融合方案 |
|------|----------|--------|---------|
| **全面性** | ✅ 全书汇总 | ❌ 单章片段 | 用 L3 做基础 |
| **准确性** | ⚠️ 可能有偏差 | ✅ 直接引文 | 优先采用 L2 |
| **易读性** | ✅ 简洁概括 | ❌ 冗长详细 | 混合呈现 |
| **故事感** | ⚠️ 可能缺乏细节 | ✅ 丰富细节 | 融入关键事件 |

### 6.2 Fallback 机制的意义

系统采用多键映射（`["intro", "summary", "description"]`），为了：
- ✅ 提高对不同 LLM 输出格式的容错性
- ✅ 支持多种 Prompt 变体
- ✅ 允许用户自定义字段名
- ✅ 灵活适应未来的扩展

### 6.3 为什么 Summary 是 Fallback？

当 LLM 没有提供某个字段时，才从 Summary 中取值。这样做是因为：
- LLM 的输出更"新鲜"，基于完整的上下文
- Summary 已经经过一次聚合，信息可能有遗漏
- 优先相信 LLM 的实时推理

## 第七部分：优化建议

### 7.1 当前设计的优势

✅ **信息融合**：兼顾全局视图和细粒度证据  
✅ **容错机制**：多键映射确保稳定输出  
✅ **模块化**：字段拆分便于定制前端展示  
✅ **可追溯**：原始数据和处理步骤都可见  

### 7.2 未来改进方向

1. **动态字段**（Dynamic Fields）
   - 根据小说类型自动调整卡片字段
   - 例：奇幻小说加"魔法能力"字段，悬疑小说加"秘密"字段

2. **多模态融合**（Multimodal Integration）
   - 集成 AI 生成的角色头像
   - 添加角色相关的背景音乐/音效
   - 关键台词的语音版本

3. **互动关系图**（Interactive Relationship Graph）
   - 在前端呈现角色之间的关系网络
   - 支持点击角色切换查看

4. **版本控制**（Version Control）
   - 记录同一角色的多个版本（章节版本）
   - 比如"第 3 章的石野" vs "第 10 章的石野"的演变

5. **RAG 增强**（RAG Enhancement）
   - 从章节文本中动态检索相关证据
   - 而非一次性加载所有数据

## 总结

角色卡是 CharPick 的**最终产品形态**，它通过一个精心设计的 Prompt 和智能的数据融合机制，将分散的提取信息（L2、L3）整合成一个**完整、可用、可交互**的角色人设。

关键特点：
- 🎯 **Prompt-Driven**：核心逻辑由 config.json 的 prompt 控制
- 🔗 **多源融合**：Summary + 逐章提取的完美结合
- 🛡️ **容错设计**：fallback 机制确保稳定性
- 📦 **模块化输出**：7 个标准字段易于前端集成
- 🔍 **可追踪**：保留所有中间步骤的数据透明性
