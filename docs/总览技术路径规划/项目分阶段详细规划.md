## 📅 CharPick 项目分阶段详细规划

### 第一阶段：底层架构与数据库拓扑（Foundation）

这个阶段的核心是解决你提到的“RAG只能存文本”和“多维层级”的问题。

#### 1. 混合型数据库架构设计
建议采用 **“关系型 (PostgreSQL) + 向量 (Milvus/Pinecone) + 对象存储 (S3/Minio)”** 的组合。
- **关系型数据库 (SQL)**：存储结构化人设（姓名、年龄、社会关系、章节索引、图片/表情包的元数据路径）。
- **向量数据库 (Vector)**：存储章节文本切片、语义化的角色性格特征、剧情摘要。
- **对象存储 (OSS)**：确保存储原始文件、生成的立绘、表情包图片。
- **关于表情包/图片存储**：RAG 并不直接存储图片像素，而是存储图片的 **CLIP Embedding（多模态向量）** 或其 **Prompt 描述**。当需要“开心”的表情包时，系统检索“角色名 + Happy”的标签，指向对象存储中的文件。

#### 2. 数据库层级规划
- **L0 原始层**：存放未处理的各格式文件信息及 OCR 后的原始长文本。
- **L1 章节解析层**：按章节切分后的文本，关联时间戳、地点标签。
- **L2 细分特征层**：你提到的“第一层提取”，包含本章人物表现、本章物品、本章事件的 JSON 化数据。
- **L3 全局画像层**：你提到的“第二层汇总”，自动聚合 L2 的碎片，形成唯一角色 ID 的最终 Persona。

---

### 第二阶段：来源获取与自动化流水线（Data Pipeline）
解决“素材哪里来”和“怎么存”的问题。
#### 1. 来源获取与爬虫层
- **脚本化采集**：开发针对主流小说网站、剧本杀平台的 Scrapy 或 Playwright 爬虫。
- **元数据对齐**：在存入数据库前，强制标注：`标题`、`作者`、`流派`、`语种`。
- **待处理队列**：利用 Redis 建立任务队列，爬取完成后自动触发后端的预处理模块。

---

### 第三阶段：人设提取工作流 (Core Workflow)
#### 1. 万能解析模块 (Parser)
- **格式自适应**：
    - **ePub/Txt**：基于 HTML 标签或正则表达式（如 `第[一二三...123]章`）切分。
    - **PDF/图片**：引入 `RapidOCR`。**建议做法**：先进行页面版面分析（Layout Analysis），剔除页眉页脚，再进行 OCR，防止杂讯污染人设。
- **逻辑分块 (Semantic Chunking)**：针对没有明确章节的作品，使用 LLM 或 BERT 模型检测“场景切换”或“时间跳跃”作为逻辑切分点。

#### 2. LLM 部署 (vLLM/Ollama)
- 可以本地也可以部署在云端服务器上面
- **模型选型**：由于 9GB 显存限制，建议使用 **Gemma-2-9B** 或 **Qwen2.5-7B** 的量化版本（Int4/AWQ）。
- **并行策略**：采用 LangChain 的 `Map-Reduce` 模式。
    - **Map**：多线程调用 LLM 同时提取不同章节的人物特征。
    - **Reduce**：由一个高上下文窗口的模型将所有章节特征融合成“全局人设”。

#### 3. 角色 Prompt 提取算法
- **维度自定义**：支持用户动态增加维度（如：增加一个“政治立场”维度）。
- **冲突处理机制**：当后期剧情与前期人设冲突时（如角色黑化），数据库需记录“性格演变轨迹”而非单纯覆盖。

---

### 第四阶段：多模态生成与应用（Application）
将提取出的“视觉特征”转化为“形象”。
- **自动化生图链**：
    - 提取到的“银发、异瞳、冷酷” -> 转化为 SD Prompt。
    - 利用 **Stable Diffusion + LoRA / IP-Adapter** 保持角色形象的一致性。
- **表情包映射**：
    - 预设 `Angry/Happy/Sad` 动作模版。
    - 将角色立绘与模版通过 AI 换脸或表情迁移技术合成。

## 🛠️ 关键技术补强建议
1. **关于表情包实现**：不要在数据库里存“死”的表情包。你应该存一个 **“角色视觉模型 (LoRA/Character Asset)”**。当需要表情包时，根据提取到的“情绪维度”实时渲染生成。
2. **前端展示载体**：
    - **酒馆AI (SillyTavern)** 是极好的前端，它支持角色卡（PNG Metadata）。你可以把你的提取工作流最终输出为一份符合酒馆格式的 `.png` 角色卡，直接导入。
3. **提取维度的可扩展性**：
    - 在数据库中使用 **JSONB 字段**（PostgreSQL 特性）。这样即使你临时想增加“三围”或者“法术列表”维度，也不需要修改数据库表结构。
