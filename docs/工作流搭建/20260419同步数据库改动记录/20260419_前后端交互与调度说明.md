# 20260419 前后端交互与调度说明

## 1. 结论先说
- 前端不要直接去执行 Python 脚本。
- 前端应该通过 HTTP 调用后端服务。
- 当前仓库里，前端最自然的连接目标是 `backend/main.py` 启动的 FastAPI 服务。
- 如果要跑完整的 L0-L2 + summary + card 链路，后端内部应该再去调 `backend/workflow_runner.py` 里的 `run_l0_to_l2_pipeline()`。

## 2. 目前仓库里两层后端入口

### 2.1 交互式 API 入口
- 文件：`backend/main.py`
- 作用：给前端提供 HTTP 接口。
- 现有接口：
  - `GET /files`
  - `POST /start-extraction`
  - `GET /logs`
- 适合：页面按钮、文件选择、日志流显示、单次提取任务。

### 2.2 完整工作流入口
- 文件：`backend/workflow_runner.py`
- 作用：串起完整工作流。
- 负责内容：
  - 读取本地源文件
  - 切章
  - L2 提取
  - summary 生成与上传
  - card 生成与上传
  - 远程 Supabase / OSS 持久化
- 适合：一键完整跑通的后端调度逻辑。

## 3. 前端应该怎么连接后端

### 3.1 推荐方式
前端只做页面和用户交互，不直接跑 Python。
前端通过 axios 或 fetch 调用后端接口，后端再去执行具体任务。

### 3.2 当前页面已经在用的连接方式
- 页面里通过 `http://localhost:8000/files` 拉取可选文件列表。
- 页面里通过 `http://localhost:8000/start-extraction` 启动任务。
- 页面里通过 `http://localhost:8000/logs` 订阅实时日志。

### 3.3 运行时顺序
1. 启动后端 FastAPI 服务。
2. 启动前端 Vite 页面。
3. 前端加载后请求文件列表。
4. 用户点击按钮后，前端把文件名、prompt、过滤开关发给后端。
5. 后端进入后台任务，持续写日志。
6. 前端通过日志流显示进度。

## 4. 前端需要调度后端的什么程序

### 4.1 如果是当前这个演示页面
前端应调度的是 `backend/main.py` 启动出来的 API 服务，不是某个独立脚本。

也就是说，前端实际触发的是这些 HTTP 入口：
- `GET /files`
- `POST /start-extraction`
- `GET /logs`

这个模式适合当前 UI 的“选择文件 + 输入 prompt + 看日志”的交互。

### 4.2 如果要做完整工作流页面
建议后端再包一层 API，让前端调用一个“完整流程”接口，由这个接口内部去执行 `run_l0_to_l2_pipeline()`。

这个完整流程内部通常要传这些参数：
- `source_path`
- `source_type`
- `source_file_id`
- `file_name`
- `filter_noise`
- `upload_chapters_to_remote`
- `upload_summary_to_remote`
- `upload_card_to_remote`
- `run_summary`
- `run_card`
- `card_character_name`
- `user_id`
- `username`
- `book_id`
- `book_title`

## 5. 推荐的职责分工

### 5.1 前端负责
- 文件选择
- 参数填写
- 提交任务
- 轮询或订阅日志
- 展示结果与错误

### 5.2 后端负责
- 接收参数
- 校验输入
- 读取本地文件
- 调用 LLM 流程
- 写本地缓存
- 上传 Supabase 和 OSS
- 返回任务状态和结果摘要

## 6. 运行命令建议

### 6.1 启动后端
在项目根目录执行：

    E:\code_environment\conda\envs\charpick\python.exe -m uvicorn backend.main:app --host 0.0.0.0 --port 8000

### 6.2 启动前端
在 frontend 目录执行：

    npm run dev

### 6.3 前端调用目标
- API 基地址：`http://localhost:8000`
- 当前页面直接依赖这个地址，不需要前端自己去启动 Python 流程。

## 7. 现阶段最重要的注意点
- 前端不要直接 import 后端 Python 模块。
- 前端不要直接执行 `backend/workflow_runner.py`。
- 完整链路的真正执行者应该是后端进程。
- 如果后续要把页面从演示模式升级到完整工作流模式，建议新增一个后端 API，把 `run_l0_to_l2_pipeline()` 封装成 HTTP 接口。

## 8. 一句话总结
前端只负责发请求和展示结果，后端 FastAPI 负责接住请求并调度真正的工作流；当前页面连接的是 `backend/main.py`，完整提取链路则应由后端内部调用 `backend/workflow_runner.py` 来完成。
