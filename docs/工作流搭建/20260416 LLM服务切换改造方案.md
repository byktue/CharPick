# LLM 服务切换改造方案

本文档说明：如果要把当前逐章提取的模型调用层改造成可切换 `Ollama / vLLM / 远程 API`，最小需要改哪些文件、`config.json` 里要加哪些字段，以及改完后如何兼容不同服务。

## 1. 先说结论

这次改造**不需要推翻整条流水线**，但需要在“模型调用层”加一层统一适配。

当前最核心的改造点是：

- `backend/llm_chapter_extraction/extractor_client.py`

上层这些文件一般不需要大改：

- `backend/llm_chapter_extraction/chapter_structured_extractor.py`
- `backend/workflow_runner.py`
- `backend/source_preprocess/pipeline.py`

也就是说，最理想的形态是：

`process_source_file(...) -> split_into_chapters(...) -> extract_chapter_structured(...) -> provider client -> Ollama / vLLM / Remote API`

## 2. 推荐改哪些文件

### 2.1 必改文件

#### 2.1.1 `backend/config.json`

用于声明当前使用哪个 LLM 服务、怎么连、用什么模型、是否需要鉴权。

#### 2.1.2 `backend/llm_chapter_extraction/extractor_client.py`

这里现在是模型调用的唯一出口，建议改成“按 provider 分发”的统一客户端。

### 2.2 建议新增文件

如果你不想把所有 provider 逻辑都堆在一个文件里，建议新增一个适配层文件，例如：

- `backend/llm_chapter_extraction/provider_clients.py`
  或
- `backend/llm_chapter_extraction/llm_client.py`

这个文件负责按 `provider` 拆分具体实现，例如：

- `ollama`
- `vllm`
- `openai_compatible`
- `custom`

### 2.3 可选调整文件

#### 2.3.1 `backend/llm_chapter_extraction/chapter_structured_extractor.py`

一般不需要大改；如果你希望它显式感知 provider，也可以把配置透传下来，但不是必须。

#### 2.3.2 测试文件

建议补充或扩展：

- `backend/test/test_llm_chapter_structured_extractor.py`
- 新增 provider 分发测试，例如：
  - Ollama 分支
  - vLLM 分支
  - OpenAI-compatible 远程 API 分支

## 3. `config.json` 建议新增哪些字段

当前 `backend/config.json` 里已经有这些字段：

- `llm.provider`
- `llm.base_url`
- `llm.model`
- `llm.temperature`
- `llm.timeout_seconds`

如果要真正兼容 Ollama / vLLM / 远程 API，建议再补下面这些字段。

### 3.1 建议新增字段

```json
{
  "llm": {
    "provider": "ollama",
    "base_url": "http://localhost:11434",
    "model": "gemma2:9b",
    "api_path": "/api/chat",
    "api_key": "",
    "temperature": 0.1,
    "timeout_seconds": 60,
    "max_retries": 2,
    "stream": false,
    "openai_compatible": false,
    "headers": {},
    "request_mode": "chat"
  }
}
```

### 3.2 每个字段的作用

- `provider`：当前选择哪个服务商
  - 例如：`ollama`、`vllm`、`openai`、`custom`

- `base_url`：服务根地址
  - 例如 Ollama：`http://localhost:11434`
  - 例如 vLLM：`http://127.0.0.1:8000`
  - 例如远程 API：`https://api.xxx.com`

- `model`：模型名
  - 例如 `gemma2:9b`
  - 例如 `Qwen/Qwen2.5-7B-Instruct`

- `api_path`：接口路径
  - Ollama 常用 `/api/chat`
  - OpenAI-compatible 常用 `/v1/chat/completions`

- `api_key`：鉴权用 token
  - 本地 Ollama 通常为空
  - 远程 API / 部分 vLLM 部署可能需要

- `temperature`：采样温度

- `timeout_seconds`：请求超时时间

- `max_retries`：失败重试次数

- `stream`：是否流式输出
  - 当前这套 JSON 提取流程建议默认 `false`

- `openai_compatible`：是否按 OpenAI Chat Completions 兼容协议调用
  - vLLM 和很多远程 API 都可以走这个模式

- `headers`：附加请求头
  - 用于自定义鉴权、租户标识等

- `request_mode`：请求模式
  - 建议先统一用 `chat`
  - 后续如果要支持 completions / embeddings 再扩展

## 4. 改完后怎么兼容 Ollama / vLLM / 远程 API

### 4.1 最小兼容思路

做一个统一的调用入口，然后按 `provider` 分发：

1. `provider = ollama`
   - 走 Ollama `/api/chat`
   - 读取 `message.content`
   - 如果需要 JSON 格式，继续沿用当前 `format: "json"` 的策略

2. `provider = vllm`
   - 如果 vLLM 开的是 OpenAI-compatible 接口，就走 `/v1/chat/completions`
   - 从 `choices[0].message.content` 取结果
   - 如果模型服务本身支持 JSON mode，可以继续要求只输出 JSON

3. `provider = openai` 或 `remote`
   - 如果远程 API 是 OpenAI-compatible，和 vLLM 走同一套解析逻辑
   - 如果不是 OpenAI-compatible，就单独写一个 adapter

### 4.2 统一的公共逻辑

不管后端是哪个服务，建议复用下面这些通用步骤：

- 统一拼 system prompt / user prompt
- 统一超时和重试
- 统一 JSON 解析和兜底清洗
- 统一错误收集
- 统一返回 `dict`

这样上层 `extract_chapter_structured(...)` 不需要知道底层到底是 Ollama、vLLM 还是远程 API。

### 4.3 推荐的解析策略

建议把返回格式统一成两种之一：

- **OpenAI-compatible**：优先推荐
  - vLLM、远程 API、OpenAI 风格服务都可以接
  - 返回格式比较统一

- **Provider adapter**：兜底方案
  - Ollama 自己一套
  - 非兼容远程 API 自己一套

### 4.4 返回值统一目标

不管走哪个 provider，最后都尽量让 `call_ollama_json(...)` 的外部调用者拿到同一种结构：

```python
{
    "chapter_info": {...},
    "plot": {...},
    "characters": {...}
}
```

也就是说，上层调用方**不要关心**底层是 Ollama 还是 vLLM。

## 5. 推荐的最小改造方案

### 5.1 第一步：只改 `extractor_client.py`

先保留现有函数名 `call_ollama_json(...)`，但内部改成：

- 读取 `config.llm.provider`
- 按 provider 分流到不同实现
- 维持外部调用签名不变

这样 `chapter_structured_extractor.py` 基本不用动。

### 5.2 第二步：先让 vLLM 和远程 API 走 OpenAI-compatible

如果你的 vLLM 和远程 API 都能提供 `/v1/chat/completions`，那兼容成本会低很多。

### 5.3 第三步：再补自定义 adapter

只有当某个远程 API 不兼容 OpenAI 协议时，再单独写 adapter。

## 6. 具体兼容建议

### 6.1 Ollama

- `provider = ollama`
- `api_path = /api/chat`
- `openai_compatible = false`

适合当前本地测试环境，改动最小。

### 6.2 vLLM

- `provider = vllm`
- `api_path = /v1/chat/completions`
- `openai_compatible = true`

如果你把 vLLM 配成 OpenAI-compatible，调用层就和远程 API 很接近。

### 6.3 远程 API

- 如果是 OpenAI-compatible：
  - `provider = openai` 或 `remote`
  - `openai_compatible = true`

- 如果不是 OpenAI-compatible：
  - `provider = custom`
  - 需要单独写请求和响应适配逻辑

## 7. 最后一句话

这次改造的核心不是“让上层知道更多 provider 细节”，而是“让上层完全不知道 provider 细节”。

只要把调用差异收口到模型客户端层，`process_source_file(...)`、`split_into_chapters(...)`、`extract_chapter_structured(...)` 这些上层流程基本都可以保持稳定。
