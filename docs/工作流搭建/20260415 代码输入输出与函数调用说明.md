## 1. 总体流程
当前主流程可以概括为：
`source_path / file_url -> 输入解析 -> 文本清洗 -> 章节切分 -> 逐章结构化提取 -> 本地缓存 / 可选远程入库`

对应主入口是：
- [backend/source_preprocess/pipeline.py](../../backend/source_preprocess/pipeline.py) 中的 `process_source_file(...)`
- [backend/llm_chapter_extraction/chapter_structured_extractor.py](../../backend/llm_chapter_extraction/chapter_structured_extractor.py) 中的 `extract_chapter_structured(...)`
- [backend/workflow_runner.py](../../backend/workflow_runner.py) 中的 `run_l0_to_l2_pipeline(...)`

## 2. 输入解析与预处理版块
### 2.1 输入
预处理入口支持两种输入方式：
- 本地文件：`source_path`
- 远程文件：`file_url`
常见附加参数：
- `source_type`: 输入类型，当前重点支持 `txt` / `md`
- `source_file_id`: 源文件唯一标识，用于本地缓存目录和文件名
- `file_name`: 源文件名
- `filter_noise`: 是否过滤前言噪声和伪章节
- `book_id` / `book_title`: 用于本地缓存目录命名和远程数据库记录

### 2.1.1 输入 → 多源文件解析 → 文本清洗 的函数链路

这一段是“章节切分”之前最基础的文本准备流程，当前对应的函数是：

- [backend/source_preprocess/parsers/__init__.py](../../backend/source_preprocess/parsers/__init__.py) 暴露的 `parse_input_to_text(...)`
- [backend/source_preprocess/parsers/router.py](../../backend/source_preprocess/parsers/router.py) 中的 `parse_input_to_text(...)`
- [backend/source_preprocess/parsers/txt_parser.py](../../backend/source_preprocess/parsers/txt_parser.py) 中的 `parse_txt(...)`、`decode_text_file(...)`、`normalize_text_file_to_utf8(...)`
- [backend/source_preprocess/text_cleaner.py](../../backend/source_preprocess/text_cleaner.py) 中的 `clean_text(...)`

推荐理解顺序是：

1. `process_source_file(...)` 先把源文件放进 `source_input/`
2. `parse_input_to_text(...)` 根据 `source_type` 选择解析器
3. 目前 `txt/md` 走 `parse_txt(...)`，读出原始文本
4. `decode_text_file(...)` / `normalize_text_file_to_utf8(...)` 负责处理编码归一化
5. `clean_text(...)` 再统一换行、空格和空白符

这一段的输出不是章节，而是“可切章的标准化纯文本”。也就是说：

- 输入：原始 txt/md 文件，或远程下载后的本地副本
- 输出：清洗后的统一文本，后续写入 `unified_format/`

### 2.2 输出

这里先只描述**预处理阶段**的输出，也就是“输入解析 + 文本清洗”这一层。

预处理阶段的直接产物主要是：
- `local_source_url`: 本地缓存后的原始文件路径
- `cache_folder`: 当前书籍缓存根目录
- `unified_format_url`: 清洗后的统一文本路径

`process_source_file(...)` 的返回值是一个字典，核心字段如下：
- `source_file_id`
- `source_type`
- `file_name`
- `local_source_url`: 本地缓存后的原始文件路径
- `cache_folder`: 当前书籍缓存根目录
- `unified_format_url`: 清洗后的统一文本路径
说明：`process_source_file(...)` 在代码实现上是一个总函数，最终返回值里确实还会继续带上 `chapters`；但从文档层级上看，`chapters` 属于下一节“章节切分版块”的输出，应放在第 3 节说明。

### 2.3 处理链路
预处理内部顺序如下：
1. 解析源文件名和缓存目录
2. 将源文件下载或复制到 `local_cache/<book_title>_<book_id>/source_input/`
3. 对 `txt/md` 执行 UTF-8 归一化
4. 调用输入解析器把原始文件读成纯文本
5. 调用清洗函数统一换行、空格和空白符
6. 写出 `unified_format/<source_file_id>.txt`
7. 调用章节切分器拆成章节
8. 写出每章 `chapter_content_text/<source_file_id>_chapter_0001.txt` 这种文件

### 2.4 关键函数
- [backend/source_preprocess/pipeline.py](../../backend/source_preprocess/pipeline.py) 的 `process_source_file(...)`
- [backend/source_preprocess/parsers/__init__.py](../../backend/source_preprocess/parsers/__init__.py) 暴露的 `parse_input_to_text(...)`
- [backend/source_preprocess/parsers/router.py](../../backend/source_preprocess/parsers/router.py) 的 `parse_input_to_text(...)`
- [backend/source_preprocess/parsers/txt_parser.py](../../backend/source_preprocess/parsers/txt_parser.py) 的 `parse_txt(...)`、`decode_text_file(...)`、`normalize_text_file_to_utf8(...)`
- [backend/source_preprocess/text_cleaner.py](../../backend/source_preprocess/text_cleaner.py) 的 `clean_text(...)`
- [backend/source_preprocess/chapter_chunker.py](../../backend/source_preprocess/chapter_chunker.py) 的 `split_into_chapters(...)`

### 2.5 需要注意
- `process_source_file(...)` 既负责“预处理”也负责“本地落盘”。
- 章节切分结果保存在本地 `chapter_content_text/`，后续逐章提取直接读取这里的文件。
- 现在的目录命名规则是 `书名_book_id`，不要再依赖旧的单书名目录结构。

## 3. 章节切分版块
### 3.1 输入
章节切分函数接收的是清洗后的整本文本：
- `text`: 单个小说的完整清洗文本
- `filter_noise`: 是否过滤前言噪声、网页残留、伪标题等

### 3.2 输出
这一层的输出就是章节列表，也就是切章后的结果。`split_into_chapters(...)` 返回 `list[dict]`，每项通常包含：

- `title`: 章节标题
- `content`: 章节正文

对应到 `process_source_file(...)` 的总返回值时，这部分会出现在 `chapters` 字段里。

### 3.3 处理逻辑
切章逻辑当前是：
1. 基于正则检测章节标题
2. 如果存在标题，先提取前言/自序
3. 对每个标题段做重复标题行去重
4. 丢弃真正空壳章节
5. 如果没有识别到标题，则回退为固定长度分块

### 3.4 关键函数
- [backend/source_preprocess/chapter_chunker.py](../../backend/source_preprocess/chapter_chunker.py) 的 `split_into_chapters(...)`
- 同文件中的内部辅助函数：
  - `_detect_heading_regex(...)`
  - `_remove_repeated_heading_lines(...)`
  - `_chapter_body_without_heading(...)`
  - `_should_drop_chapter(...)`

### 3.5 注意事项
- 章节切分当前是“先识别标题，再按标题截段”。
- 切章输出中的 `content` 是后续 LLM 提取的直接输入。
- 如果文本里有大量前言、广告、链接，建议打开 `filter_noise=True`。

## 4. 逐章结构化提取版块
### 4.1 输入
`extract_chapter_structured(...)` 接收的核心输入是：
- `chapter_title`: 当前章节标题
- `chapter_text`: 当前章节正文
- `enabled_dimensions`: 可选，只提取部分维度
### 4.2 输出
函数返回一个结构化字典，典型字段有：
- `chapter_title`
- `chapter_info`
- `plot`
- `characters`
- `items`
- `world`
- `errors`：可选，记录某个维度提取失败的错误信息
### 4.3 提取逻辑
当前逐章提取不是一次性大 Prompt，而是按维度循环调用：
1. 读取 `backend/config.json` 中的 LLM 配置和 prompts
2. 组装 `chapter_title + chapter_text[:6000]` 作为基础输入
3. 按维度调用 `call_ollama_json(...)`
4. 清理空字段
5. 把各维度结果合并成一个 JSON

### 4.4 关键函数
- [backend/llm_chapter_extraction/chapter_structured_extractor.py](../../backend/llm_chapter_extraction/chapter_structured_extractor.py) 的 `extract_chapter_structured(...)`
- [backend/llm_chapter_extraction/extractor_client.py](../../backend/llm_chapter_extraction/extractor_client.py) 的 `call_ollama_json(...)`

### 4.5 需要注意
- 这个函数本身不直接控制 GPU，它只通过 HTTP 请求调用 Ollama。
- GPU 是否被使用，由 Ollama 容器和宿主机环境决定。
- 当前实现中，每个章节会按维度依次提取，因此总调用次数 = 章节数 × 维度数。

## 5. 一体化调度版块
### 5.1 输入
`run_l0_to_l2_pipeline(...)` 负责把整个流程串起来，常用输入如下：
- `source_path` / `file_url`
- `source_type`
- `source_file_id`
- `file_name`
- `filter_noise`
- `enabled_dimensions`
- `upload_chapters_to_remote`
- `remote_db_url`
- `user_id` / `username`
- `book_id` / `book_title`

### 5.2 输出
返回值是一个字典，核心字段如下：
- `source_file_row`
- `chapter_rows`
- `chapter_extraction_rows`
- `chapter_extraction_cache_dir`
- `remote_persist`

### 5.3 处理逻辑
一体化流程顺序是：
1. 调用 `process_source_file(...)`
2. 遍历 `chapters`
3. 对每章调用 `extract_chapter_structured(...)`
4. 将 L2 JSON 写入本地 `extraction_json/`
5. 视参数决定是否调用远程入库

### 5.4 关键函数

- [backend/workflow_runner.py](../../backend/workflow_runner.py) 的 `run_l0_to_l2_pipeline(...)`

### 5.5 需要注意

- 当前 L2 JSON 本体默认先写本地，不默认上传远程。
- 只有当 `upload_chapters_to_remote=True` 时，才会走 `persist_l0_l1_rows(...)` 和 `persist_l2_rows(...)`。
- 如果只想验证章节切分，就只调用 `process_source_file(...)`。
- 如果只想验证提取，可以直接遍历 `chapter_content_text/` 里的章节文件调用 `extract_chapter_structured(...)`。

## 6. 远程数据库入库版块

### 6.1 输入

远程入库函数主要吃两类数据：

- L0/L1 数据：用户、书籍、源文件、章节
- L2 数据：章节提取结果 JSON 路径

### 6.2 输出

返回值主要是入库统计信息，例如：

- `user_id`
- `book_id`
- `source_file_id`
- `uploaded_chapter_count`
- `uploaded_extraction_count`

### 6.3 处理逻辑

- `persist_l0_l1_rows(...)` 写入 `users`、`books`、`source_files`、`chapters`
- `persist_l2_rows(...)` 写入 `chapter_extractions`
- 数据库连接支持 Supabase 主连接串和 pooler 回退

### 6.4 关键函数

- [backend/remote_persistence.py](../../backend/remote_persistence.py) 的 `persist_l0_l1_rows(...)`
- [backend/remote_persistence.py](../../backend/remote_persistence.py) 的 `persist_l2_rows(...)`

### 6.5 需要注意

- 目前写入远程的只是路径/URL，不是大块正文直接写库。
- 如果你本地只跑测试，不要开启 `upload_chapters_to_remote`。

## 7. 推荐调用方式

### 7.1 只跑切章
适合先验证输入解析、清洗和章节拆分是否正确。
调用：
- `process_source_file(...)`


### 7.2 只跑切章后的提取
适合已经有 `chapter_content_text/`，只想验证 LLM 提取是否正常。
调用：
- `extract_chapter_structured(...)`


### 7.3 跑完整链路

适合从源文件直接验证全部流程。

调用：

- `run_l0_to_l2_pipeline(...)`

## 9. 当前实现约定

1. 预处理只负责把文本整理成可供后续处理的标准输入。
2. 切章输出保存在 `chapter_content_text/`，提取输出保存在 `extraction_json/`。
3. 逐章提取是“按章节、按维度”循环调用，不是一次性全书大合并。
4. Python 代码不直接控制 GPU，GPU 是否生效取决于 Ollama 服务本身。
5. 当前 L2 JSON 默认先落本地，便于你先检查结果再决定是否接远程存储。
