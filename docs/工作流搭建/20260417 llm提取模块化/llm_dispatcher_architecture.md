# CharPick LLM 调度架构说明

本文档说明当前项目中独立 LLM 调度模块的推荐架构。核心原则只有一句话：**所有语义变化尽量交给 prompt 控制，代码只负责调度、检索、拼接、执行和落盘。**

## 1. 设计目标

这个调度架构要同时满足三件事：

- 支持 `chapter_prompts`、`summary_prompts`、`card_prompts` 三类任务。
- 支持不同任务使用不同模型、不同参数、不同 prompt。
- 支持后续个性化扩展，而不把语义结构锁死在固定中间层里。

## 2. 设计原则

### 2.1 prompt 负责语义，代码负责流程

不应该把“最终要抽什么”写死在代码里。更合理的做法是：

- `chapter_prompts` 决定逐章提取什么。
- `summary_prompts` 决定整书汇总时关注什么。
- `card_prompts` 决定展示卡要输出什么。

代码层只做下面这些通用动作：

- 读取逐章 JSON。
- 按规则找出每章中与目标相关的片段。
- 拼接上下文。
- 调用 LLM。
- 保存 JSON / Markdown 结果。

### 2.2 不做固定语义中间层

不建议再加一个固定 schema 的“标准化中间层”，例如强行统一成：

- `chapter_summary`
- `key_events`
- `characters`
- `items`
- `world_locations`

这种做法短期看起来整齐，长期会限制你后面做个性化扩展。

正确方式是：

- 保留原始逐章 JSON。
- 由 prompt 决定要从每章里取哪些部分。
- 调度器只负责把相关片段重新拼起来。

### 2.3 不是“按章节召回”，而是“按章节内片段召回”

这里要特别强调：

- 不应该粗暴地按章节整块召回。
- 而应该按规则找回**每个章节中对应的部分**。

也就是说，调度器不是在问“这一章要不要用”，而是在问：

- 这一章里的角色片段在哪里？
- 这一章里的物品片段在哪里？
- 这一章里的剧情片段在哪里？
- 这一章里的地点 / 世界观片段在哪里？

然后只把这些相关片段拿去拼接上下文。

## 3. 总体链路

推荐的整体链路如下：

1. 先完成 chapter 级别提取，得到逐章 JSON。
2. summary / card 调度器读取这些逐章 JSON。
3. 调度器根据当前 prompt 的目标，提取每章中相关字段或片段。
4. 将这些片段拼成目标任务上下文。
5. 调用对应模型生成 summary 或 card。
6. 将最终结果写入本地缓存目录。

## 4. 目录结构建议

### 4.1 逐章提取结果

- `local_cache/<book>/extraction_json/`

用于保存每章的提取结果，每个文件对应一章。

### 4.2 summary 输出

- `local_cache/<book>/summary/summary_json/`
- `local_cache/<book>/summary/summary_md/`

用于保存总表提取结果及前端展示用 Markdown。

### 4.3 card 输出

- `local_cache/<book>/card/card_json/`
- `local_cache/<book>/card/card_md/`

用于保存展示卡结果及前端展示用 Markdown。

## 5. 三类 prompt 的职责

### 5.1 chapter_prompts

职责是对单章内容做结构化提取。

典型用途：

- 章节信息
- 剧情概要
- 角色行为 / 对白
- 物品
- 地点 / 世界观

### 5.2 summary_prompts

职责是定义整书汇总任务，但**不是**定义固定字段 schema。

它应该告诉调度器：

- 需要关注什么主题。
- 需要从逐章 JSON 的哪些部分找材料。
- 需要最终生成什么总表。

例如：

- 角色总表
- 物品总表
- 剧情时间线
- 世界观地图表

### 5.3 card_prompts

职责是定义展示卡输出目标。

它可以进一步针对前端展示优化：

- 角色简介
- 外形特征
- 性格标签
- 关键经历
- 常用台词
- 关系摘要

card 目前的输入方式是：前端先从 summary 中看到可选人物，再把选中的角色名字传给后端；后端再从逐章 JSON 里找出与该角色相关的片段，拼接后生成角色卡。

## 6. 推荐实现方式

### 6.1 代码层只做“片段级拼接”

调度器读取每章 JSON 后，不要把整章全文直接扔给 summary 模型，而是先按 prompt 规则挑片段，例如：

- 角色总表：取角色相关片段、姓名、行为、对白、关系。
- 物品总表：取物品名、描述、出现位置、所属角色。
- 时间线：取章节顺序、剧情摘要、关键事件。
- 世界观：取地点、设定、势力、场景描述。

### 6.2 prompt 决定“找什么”，不是“硬编码字段”

后续如果你要做个性化延申，比如：

- 某角色成长轨迹
- 某关系网
- 某道具演化史
- 某场景氛围卡

都应该通过新增 prompt 解决，而不是改固定中间层。

### 6.3 调度器保持可插拔

调度器应该支持：

- 换模型
- 换 prompt
- 换输入目录
- 换输出目录
- 增加新的任务类型

但不应该把业务语义写死。

## 7. 当前推荐理解

可以把它理解成三层：

- **数据层**：逐章 JSON。
- **调度层**：按 prompt 找回每章中对应片段并拼接。
- **生成层**：LLM 根据 prompt + 拼接上下文生成 summary 或 card。

这里的关键不是“先做一个固定中间层”，而是：

- 保留原始信息。
- 用 prompt 控制召回逻辑。
- 用调度层把章节内相关部分拼起来。

## 8. 适合后续演进的方向

如果后续继续扩展，建议只增加新的 prompt 族，而不是重做架构：

- `character_growth_prompts`
- `relationship_prompts`
- `artifact_evolution_prompts`
- `scene_atmosphere_prompts`

这样你仍然可以保持：

- prompt 控语义
- 代码控流程
- 数据保持原样可追溯
