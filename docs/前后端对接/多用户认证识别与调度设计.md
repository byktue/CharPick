# 多用户认证识别与调度设计

## 目标
- 保证“谁触发提取，数据就写到谁名下”。
- 防止跨用户读取/调度别人的书籍。
- 在调度链路中传递 user_id，实现用户级隔离。

## 认证来源
前端登录后保存 auth_token（Bearer Token），并在请求头中传给后端：
- Authorization: Bearer <token>

后端在 /api/v1/extract、/api/v1/characters 等接口中执行 token 校验。

## 用户识别流程
1. 后端解析 Authorization 头。
2. 从 auth_sessions 表按 access_token_jti 查询会话。
3. 校验 revoked_at / expires_at（未吊销且未过期）。
4. 关联 users 表得到 user_id、username。

若任一步失败：
- 返回 401（登录态无效或过期）。

## 资源授权规则

### 书籍授权
在触发调度时：
- 以 user_id + book_id 查询 books 表。
- 不存在则返回 404（不泄露他人资源）。

### 角色授权
在角色查询时：
- card 表始终附加 user_id 过滤。
- 仅返回当前用户可见的数据。

## 调度上下文注入
调度任务实际执行 run_l0_to_l2_pipeline 时注入：
- user_id
- username
- book_id
- book_title

并设置：
- upload_chapters_to_remote=True
- upload_summary_to_remote=True（可覆盖）
- upload_card_to_remote=True（可覆盖）

这样写入远程数据库时自然带用户归属。

## 多用户并发调度

### 任务隔离
- 每个调度请求生成独立 task_id（dispatch_xxx）。
- 后端以内存任务表记录每个 task_id 的进度，不互相覆盖。

### 数据隔离
- 远程持久化以 user_id + book_id 为主键维度。
- 前端读取远程表时也按 user_id 过滤（已有 Supabase 查询策略）。

## 前后端协同关系

### 后端负责
- token 校验
- user_id 解析
- book 归属校验
- 调度执行和远程写入

### 前端负责
- 保存和携带 auth_token
- 调用 /api/v1/extract 触发任务
- 从远程数据库按用户维度读取任务、summary、card

## 安全建议
1. 生产环境收紧 CORS（避免 allow_origins=*）。
2. 对 /api/v1/extract 增加频率限制（按 user_id）。
3. task_id 状态查询可增加“仅创建者可读”的校验。
4. 对敏感错误信息做脱敏，避免泄露连接配置。

## 失败场景与回滚
- 调度失败时任务状态记为 failed。
- 远程层会写失败信息到 books/extraction_tasks。
- 前端可从 extraction_tasks 和 books.err_message 读取错误原因。
