# 后端统一端口设计

## 设计目标
- 以单一服务端口承载调度、健康检测、角色查询、聊天代理。
- 对前端暴露稳定 API 前缀：/api/v1。
- 保持旧接口可用，避免一次性迁移风险。

## 统一端口
- 基础地址：http://127.0.0.1:8000
- 推荐 API 前缀：/api/v1

## 端口与路由总览

### 健康检查
- GET /health
- GET /api/v1/health

用途：
- 设置页“后端连通检测”。

### 调度接口
- POST /api/v1/extract

用途：
- 前端触发 L0->L2->summary->card 的全链路处理。
- 后端在后台执行并上传远程数据库。

关键参数：
- book_id: 书籍 ID（必填）
- roles: 角色 ID 列表（可空）
- card_character_name: 角色卡目标角色名（可选）
- run_summary/run_card: 是否执行汇总和角色卡
- upload_summary_to_remote/upload_card_to_remote: 是否上传远程
- source_file_id/source_type/file_url: 可覆盖自动推断

### 调度状态查询
- GET /api/v1/tasks/{task_id}

用途：
- 查询后端调度任务的内存态进度（queued/running/completed/failed）。
- 作为前端轮询调度状态的补充能力。

### 角色查询
- GET /api/v1/characters
- GET /api/v1/characters/{character_id}

用途：
- 直接由后端按 user_id 范围返回 card 表数据。
- 对接角色详情页或后端代理查询场景。

### 聊天代理
- POST /chat

用途：
- 前端 chat 页通过统一端口进行模型调用。
- 后端根据 backend/config.json 中 llm.provider 路由到：
  - remote_api（OpenAI 兼容）
  - ollama（本地模型）

## 请求流设计（调度）
1. 前端调用 POST /api/v1/extract。
2. 后端验证 Authorization -> user_id。
3. 后端按 user_id + book_id 查 books，拿 book_file_url。
4. 后台任务调用 run_l0_to_l2_pipeline：
   - 下载/预处理
   - 章节提取
   - summary 汇总
   - card 生成
   - 远程入库（books/extraction_tasks/chapter_extractions/summary/card）
5. 返回 task_id，前端可查 /api/v1/tasks/{task_id}，也可直接看远程表变化。

## 错误处理设计
- 401：Authorization 缺失或 token 无效。
- 404：book_id 不存在或不属于当前用户。
- 400：book_file_url 为空等请求参数错误。
- 502：上游模型调用失败（chat 代理场景）。

## 兼容策略
保留旧接口：
- GET /files
- GET /logs
- POST /start-extraction

理由：
- 避免老脚本立即失效。
- 逐步迁移到 /api/v1/extract。

## 建议的前端配置
- settings.backend_url 默认：http://127.0.0.1:8000
- 业务 API：通过 apiV1Base 自动拼接到 /api/v1
- 聊天 API：使用 /chat（非 OpenAI 直连模式）
