# 数据库总体框架

## 1. 设计目标

CharPick 的数据库不是单一的“角色表”，而是一个围绕“原始文件 -> 章节切分 -> 章节提取 -> 角色汇总 -> 前端查询”的分层数据底座。

它要同时解决四件事：

1. 把 epub、txt、pdf、md、图片 OCR 等输入统一沉淀为可追溯数据。
2. 让章节级提取、角色卡生成、剧情时间线、物品与世界观信息能够结构化存储。
3. 支持后续的向量检索、图片资源管理、任务进度管理和版本回溯。
4. 从一开始就具备用户隔离能力，保证不同账户的数据互不串用。

## 2. 总体原则

### 2.1 单库优先

当前阶段不建议按用户拆数据库，而是采用“单个 PostgreSQL 实例 + 用户归属字段 + Token 鉴权”的方式。

这样做的好处是：

1. 部署简单，适合 MVP。
2. 查询链路短，列表接口可以直接按 `user_id` 过滤。
3. 后续如果要做协作、共享、权限组，只需要在现有模型上扩展，不需要重构整套存储。

### 2.2 分层建模

数据库需要按数据成熟度分层，而不是把所有信息都塞进一张大表里。

推荐分为四层：

1. 原始层：保存文件来源、导入元数据、OCR 文本、对象存储地址。
2. 章节层：保存章节切分后的文本、章节编号、章节范围。
3. 提取层：保存每章抽取出的结构化结果，通常是 JSONB。
4. 汇总层：保存书级角色库、物品库、剧情线、世界观和角色版本。

### 2.3 业务表必须可追溯到用户

所有核心业务表都应带有 `user_id`：

- `books.user_id`
- `source_files.user_id`
- `chapters.user_id`
- `chapter_extractions.user_id`
- `characters.user_id`
- `character_versions.user_id`
- `items.user_id`
- `storyline_events.user_id`
- `world_locations.user_id`
- `media_assets.user_id`
- `extraction_tasks.user_id`

这样 `/api/v1/books`、`/api/v1/characters` 这类列表接口就能直接按当前用户做过滤。

## 3. 系统分层

### 3.1 接入层

接入层负责和前端交互，主要包括：

- 用户注册和登录
- 上传文件
- 查询书籍、角色、章节、时间线
- 获取任务进度
- 导出角色卡或相关资源

接入层的关键点是：所有请求都要先识别当前用户，再进入业务查询。

### 3.2 业务服务层

业务服务层负责把“一个上传动作”拆成多个稳定步骤：

- 文件解析
- 章节切分
- 章节提取
- 角色汇总
- 向量生成
- 任务状态更新

这层建议由 FastAPI 配合独立的服务函数或任务流程实现，避免把复杂逻辑塞进路由层。

### 3.3 数据访问层

数据访问层统一处理：

- ORM 模型
- 数据库连接
- 查询条件拼装
- 事务提交与回滚
- 迁移管理

推荐使用 SQLAlchemy + Alembic 作为标准方案。

### 3.4 存储层

存储层由几类组件组成：

- PostgreSQL：主业务数据库
- pgvector：向量检索能力
- MinIO：原始文件、图片、封面、角色图存储
- Redis：任务进度、短期状态、异步调度

## 4. 四层数据结构

### 4.1 L0 原始层

这一层保存“原始内容是什么、从哪里来、什么时候导入”的信息。

建议保存：

- `user_id`
- `book_id`
- `source_type`
- `file_name`
- `file_path` 或 `object_url`
- `file_hash`
- `title`
- `author`
- `import_status`
- `created_at`

如果是图片或扫描件，还应保存 OCR 文本和版面分析结果。

### 4.2 L1 章节层

这一层对应书籍切分后的最小阅读单元。

建议保存：

- `user_id`
- `book_id`
- `chapter_no`
- `chapter_title`
- `chapter_range`
- `story_time`
- `content_text`
- `word_count`
- `source_file_id`

这一层尽量保持纯净，只保存可直接回放的章节内容，不混入过多推断结果。

### 4.3 L2 提取层

这一层保存每章提取出来的结构化结果，通常使用 JSONB。

建议保存：

- `user_id`
- `book_id`
- `chapter_id`
- `extractor_type`
- `extraction_json`
- `embedding_vector`
- `prompt_version`
- `model_name`
- `confidence`

这层是章节分析和角色汇总之间的桥梁，便于回溯“结论是怎么来的”。

### 4.4 L3 汇总层

这一层是前端最常查询的结果层。

建议保存：

- `characters`：角色总表
- `character_versions`：角色版本记录
- `items`：关键物品
- `storyline_events`：剧情与时间线
- `world_locations`：地点与世界观
- `media_assets`：图片、封面、表情包、立绘

汇总层同样必须按 `user_id` 过滤，避免跨用户读取。

## 6. 业务流转路径

### 6.1 上传流程

1. 用户登录并获取 Token。
2. 前端上传文件。
3. 后端从 Token 解析出 `user_id`。
4. 后端创建 `books`、`source_files` 和任务记录。
5. 原始文件保存到 MinIO，元数据写入 PostgreSQL。

### 6.2 提取流程

1. 格式解析器读取原始文件。
2. 系统切分章节并写入 `chapters`。
3. 每章调用提取器生成结构化 JSON。
4. 提取结果写入 `chapter_extractions`。
5. 同步写入向量数据和任务进度。

### 6.3 汇总流程

1. 系统把章节级结果聚合成角色、物品、剧情、地点。
2. 汇总结果写入 `characters`、`items`、`storyline_events`、`world_locations`。
3. 角色卡按版本写入 `character_versions`。
4. 前端通过用户作用域接口读取最终结果。

## 7. 查询与权限

### 7.1 查询原则

所有列表接口都必须先按 `user_id` 过滤，再按业务条件过滤。

例如：

- `/api/v1/books` 只查当前用户的书籍。
- `/api/v1/characters` 只查当前用户的角色库。
- `/api/v1/books/{book_id}/roles` 只能读取当前用户名下的书。

### 7.2 权限边界

后端不应该信任前端传来的 `user_id`。

正确做法是：

1. 从 `Authorization: Bearer <Token>` 中解析当前用户。
2. 用解析出的 `user_id` 作为所有业务查询的过滤条件。
3. 只有在管理员或协作场景下，才额外引入成员表和共享权限表。

## 8. 推荐技术组合

当前阶段建议使用：

- FastAPI：接口层
- PostgreSQL：主业务库
- SQLAlchemy：ORM
- Alembic：迁移管理
- pgvector：向量检索
- MinIO：对象存储
- Redis：任务状态和缓存
- JWT / OAuth2：认证鉴权

这套组合的特点是部署成本低、扩展路径清晰，适合先把 MVP 跑通。

## 9. 演进路线

### 9.1 MVP 阶段

先做最小闭环：

1. 用户注册、登录和 Token 识别。
2. books、source_files、chapters、chapter_extractions 四层基础表。
3. 角色、物品、剧情、地点的汇总入库。
4. `/api/v1/books` 和 `/api/v1/characters` 按 `user_id` 过滤。

### 9.2 增强阶段

在 MVP 稳定后补充：

1. `character_versions` 版本化。
2. `media_assets` 图像元数据。
3. `auth_sessions` 刷新与设备管理。
4. pgvector 检索和更细的任务日志。

### 9.3 协作阶段

如果后续要多人共用同一本书，再扩展：

- 成员表
- 共享权限表
- 书籍协作角色表

这样可以在不破坏现有结构的前提下升级成多人协作系统。
