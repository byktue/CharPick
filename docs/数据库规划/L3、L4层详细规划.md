L3为数据汇总层
L4则为卡层

数据汇总版块，可以用通用模块进行展示，即summary表
角色卡展示也可用通用模块，即card表（如果之后要做环境信息提取，或者物品详细信息提取，都可以套用这个模块）

### summary（总表模块）
- id (PK) # 表唯一 ID M
- user_id (FK → users) # 所属用户 M
- book_id (FK → books) # 所属书籍 M
- type # 表类型 M
- name # 表名称
- json_url # json格式内容地址，具体文件存储于oss中 M （用于具体后端调用）
- content_url # 详细内容地址，**用于前端展示**，具体文件存储于oss中 M
- created_at # 创建时间 M
- updated_at # 更新时间 M

| type             | name    |
| ---------------- | ------- |
| characters       | 角色总表    |
| items            | 物品总表    |
| storyline_events | 剧情时间线   |
| world_locations  | 世界观/地点表 |

### card （展示卡模块）
- id (PK) # 唯一ID M
- user_id (FK → users) # 所属用户 M
- book_id (FK → books) # 所属书籍 M
- type # 类型 M
- name # 名称 M
- intro # 简介
- image_url # 图像地址，具体文件存储于oss中 
- json_url # json格式内容地址，具体文件存储于oss中 M （用于具体后端调用）
- content_url # 该卡详细内容地址，具体文件存储于oss中 M
- created_at # 创建时间 M
- updated_at # 更新时间 M

### 改动说明
删表 chapters, characters, character_versions, items, storyline_events, world_locations
新建 summary, card

解释：
- chapters存储的是章节拆分后的文件，这一步处理速度很快，没必要占用数据库资源，可以本地缓存，需要进行云端存储的主要是chapter_extractions 所以保留chapter_extractions
- characters, character_versions, items, storyline_events, world_locations 都可以用summary, card进行完美替换，并且之后新增模块会更加方便
