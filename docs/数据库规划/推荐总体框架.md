# 推荐总体框架

## 1. 设计目标

CharPick 的数据库建议采用“单库 + 用户归属字段 + 分层存储”的方式，而不是按用户拆分多套数据库。

核心目标有三个：

1. 统一承接 epub、txt、pdf、md、图片 OCR 等输入来源。
2. 支持章节切分、章节提取、角色汇总、剧情整理、世界观沉淀。
3. 为前端展示、角色卡导出、向量检索、图片资源管理和后续对话调用提供稳定底座。

## 2. 结论先行：是否需要用户管理

### 2.1 结论

需要。

推荐方案是：**单个 PostgreSQL + users 表 + 业务表 user_id + Token 鉴权**。

### 2.2 原因

1. 接口已经要求使用 Bearer Token，后端需要从 Token 中识别当前用户。
2. books、characters 这类列表接口天然需要按 user_id 过滤。
3. 把 user_id 放在主业务表里，比单独维护“用户-资源映射表”更直接，也更利于索引。
4. 未来如果要做协作和共享，可以在现有模型上补成员表和权限表，不需要重构整套数据库。

### 2.3 单用户起步建议

即使现在只有一个人使用，也建议提前预留：

- users 表
- user_id 归属字段
- Token 鉴权
- 默认用户或默认租户

这样以后从单用户扩展到多人时，只需要补登录、注册和权限控制，不需要推倒重来。

## 3. 推荐总体架构

### 3.1 四层存储

| 层级 | 作用 | 建议存储 |
| --- | --- | --- |
| L0 原始层 | 原始文件、OCR 结果、导入元数据 | PostgreSQL + MinIO |
| L1 章节层 | 按章节切分后的纯文本与章节定位 | PostgreSQL |
| L2 提取层 | 每章提取出的角色、剧情、地点、物品、时间线 | PostgreSQL JSONB |
| L3 汇总层 | 书级角色总表、世界观总表、角色卡、动态画像 | PostgreSQL + pgvector |

### 3.2 资源分工

| 组件         | 作用                       | 是否必需 |
| ---------- | ------------------------ | ---- |
| PostgreSQL | 主业务库，保存书籍、章节、角色、汇总、关系    | 必需   |
| pgvector   | 保存文本切片或摘要向量，用于检索         | 推荐   |
| MinIO / S3 | 保存原始文件、封面、OCR 图片、角色图、表情包 | 推荐   |
| Redis      | 任务队列、进度状态、短期缓存           | 推荐   |

### 3.3 认证与权限层

在数据库和接口之间补一层认证与权限控制，负责：

- 用户注册、登录、Token 签发
- 当前用户身份识别
- 书籍和角色库的归属过滤
- 后续协作与共享权限扩展

### 3.4 核心业务表总览

推荐把业务表分成“用户表、项目表、过程表、结果表、资源表、任务表”六类，方便你后续拆模块。

| 类别  | 代表表                                                                  | 作用                      |
| --- | -------------------------------------------------------------------- | ----------------------- |
| 用户表 | users、auth_sessions                                                  | 管理账号、登录态、Token、会话失效     |
| 项目表 | books                                                                | 作为一本书、一个角色库或一个剧本项目的主入口  |
| 过程表 | source_files、chapters、chapter_extractions                            | 记录文件来源、章节切分和章节级提取过程     |
| 结果表 | characters、character_versions、items、storyline_events、world_locations | 保存角色、版本、物品、剧情、地点等业务结果   |
| 资源表 | media_assets                                                         | 保存封面、立绘、表情包、图片等多模态资源元数据 |
| 任务表 | extraction_tasks、task_logs                                           | 记录导入、解析、提取、汇总等异步任务与日志   |

### 3.5 四层存储与业务表对应关系

四层存储不是纯技术分层，它本身就是业务表的落位方式。

| 层级     | 对应业务表                                                                             | 这层存什么                      |
| ------ | --------------------------------------------------------------------------------- | -------------------------- |
| L0 原始层 | books、source_files                                                                | 书籍主入口、原始文件、导入元数据、OCR 来源    |
| L1 章节层 | chapters                                                                          | 切分后的章节文本、章节编号、章节范围、章节定位    |
| L2 提取层 | chapter_extractions                                                               | 每章抽出的角色、物品、剧情、地点、时间线 JSONB |
| L3 汇总层 | characters、character_versions、items、storyline_events、world_locations、media_assets | 面向前端和角色库的最终业务结果            |
| 通用支撑   | users、auth_sessions、extraction_tasks、task_logs                                    | 用户身份、登录态、任务过程、任务日志         |

简单理解就是：
1. `books` 和 `source_files` 负责“这本书从哪来”。
2. `chapters` 负责“它被切成了什么章节”。
3. `chapter_extractions` 负责“每章提取出了什么结构化结果”。
4. `characters` 等汇总表负责“最终给前端和角色库看的结果”。
5. `users`、`auth_sessions`、`extraction_tasks`、`task_logs` 是贯穿全流程的支撑表。

## 4. 数据分层设计
### 4.1 L0 原始层
这一层负责保存“文件从哪来、原始是什么、版本是什么”。
建议字段：
- user_id
- book_id
- source_type
- file_name
- file_path 或 object_url
- file_hash
- title
- author
- import_status
- created_at
如果是图片来源，需要额外保存 OCR 文本、版面分析结果、图片对象地址。

### 4.2 L1 章节层

这一层对应章节切分结果，也是后续提取的最小单元。

建议字段：

- user_id
- book_id
- chapter_no
- chapter_title
- chapter_range
- story_time
- content_text
- word_count
- source_file_id
- parser_type

这一层要尽量保持纯净，不要混太多推断结果。

### 4.3 L2 提取层

这一层存的是每章提取出的结构化结果，建议使用 JSONB。

建议字段：

- user_id
- book_id
- chapter_id
- extractor_type
- extraction_json
- embedding_vector
- prompt_version
- model_name
- confidence

主要内容包括：

- 本章角色
- 本章关键物品
- 本章剧情事件
- 本章地点和背景
- 本章时间线
- 提取置信度与来源片段

### 4.4 L3 汇总层

这一层是前端最常展示的层，也对应角色总表、剧情时间线、世界观总表、角色卡。

建议保存：

- characters：角色总表
- character_versions：角色版本记录
- items：关键物品
- storyline_events：剧情与时间线
- world_locations：地点与世界观
- media_assets：图片、封面、表情包、立绘

汇总层数据在查询时必须先校验 user_id，避免跨用户读取。

## 6. 业务流转路径

### 6.1 上传流程

1. 用户登录并获取 Token。
2. 前端上传文件。
3. 后端从 Token 解析出 user_id。
4. 后端创建 books、source_files 和任务记录。
5. 原始文件保存到 MinIO，元数据写入 PostgreSQL。

### 6.2 提取流程

1. 格式解析器读取原始文件。
2. 系统切分章节并写入 chapters。
3. 每章调用提取器生成结构化 JSON。
4. 提取结果写入 chapter_extractions。
5. 同步写入向量数据和任务进度。

### 6.3 汇总流程

1. 系统把章节级结果聚合成角色、物品、剧情、地点。
2. 汇总结果写入 characters、items、storyline_events、world_locations。
3. 角色卡按版本写入 character_versions。
4. 前端通过用户作用域接口读取最终结果。

## 7. 查询与权限

### 7.1 查询原则

所有列表接口都必须先按 user_id 过滤，再按业务条件过滤。

例如：

- /api/v1/books 只查当前用户的书籍。
- /api/v1/characters 只查当前用户的角色库。
- /api/v1/books/{book_id}/roles 只能读取当前用户名下的书。

### 7.2 权限边界

后端不应该信任前端传来的 user_id。

正确做法是：

1. 从 Authorization: Bearer <Token> 中解析当前用户。
2. 用解析出的 user_id 作为所有业务查询的过滤条件。
3. 只有在管理员或协作场景下，才额外引入成员表和共享权限表。

## 8. 推荐技术组合

当前阶段建议使用：

- FastAPI：接口层
- PostgreSQL：主业务库
- SQLAlchemy：ORM
- Alembic：迁移管理
- pgvector：向量检索
- MinIO：对象存储
- Redis：任务状态和缓存
- JWT / OAuth2：认证鉴权

这套组合的特点是部署成本低、扩展路径清晰，适合先把 MVP 跑通。

## 9. 演进路线

### 9.1 MVP 阶段

先做最小闭环：

1. 用户注册、登录和 Token 识别。
2. books、source_files、chapters、chapter_extractions 四层基础表。
3. 角色、物品、剧情、地点的汇总入库。
4. /api/v1/books 和 /api/v1/characters 按 user_id 过滤。

### 9.2 增强阶段

在 MVP 稳定后补充：

1. character_versions 版本化。
2. media_assets 图像元数据。
3. auth_sessions 刷新与设备管理。
4. pgvector 检索和更细的任务日志。

### 9.3 协作阶段

如果后续要多人共用同一本书，再扩展：

- 成员表
- 共享权限表
- 书籍协作角色表

这样可以在不破坏现有结构的前提下升级成多人协作系统。
