# 🔐 0. 用户认证与管理 (Auth & Users)
**功能**：处理用户的注册、登录鉴权以及基础信息获取。后续所有需要区分用户的接口，均需在 HTTP Header 中携带 `Authorization: Bearer <Token>`。
### 用户注册
- **方法**：POST
- **路径**：`/api/v1/auth/register`
- **主要输入参数** (JSON)：
  - `username`: str
  - `password`: str
  - `email`: str (可选)
- **业务逻辑**：校验用户名是否重复，对密码进行哈希加密后存入数据库。

**输出示例（JSON）**
```json
{
  "user_id": "u_8848",
  "username": "author_01",
  "message": "User registered successfully"
}
```


---
### 用户登录 (获取 Token)
- **方法**：POST
- **路径**：`/api/v1/auth/login`
- **主要输入参数** (JSON 或 Form)：
  - `username`: str
  - `password`: str
- **业务逻辑**：校验账号密码，验证通过后签发 Access Token（如 JWT）。

**输出示例（JSON）**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6Ikp...",
  "token_type": "bearer",
  "expires_in": 86400
}
```


---
### 获取当前用户信息
- **方法**：GET
- **路径**：`/api/v1/users/me`
- **说明**：主要用于前端在初始化时验证 Token 是否有效，并拉取用户基本信息。
- **依赖**：需要校验 Token。

**输出示例（JSON）**
```json
{
  "user_id": "u_8848",
  "username": "author_01",
  "created_at": "2026-04-07T10:00:00Z"
}
```

# 📡 1. 上传与任务处理接口 (Task & Upload)
**功能**：处理文件上传、启动 L0-L1 解析，并提供进度查询。
> 注：所有接口均需在 Header 中携带 `Authorization: Bearer <Token>`，后端从 Token 解析 `user_id` 做数据隔离与权限校验。
### 文件上传
- **方法**：POST
- **路径**：`/api/v1/upload`
- **主要输入参数**：
  - `file`: Multipart
  - `book_name`: str（可选）
- **业务逻辑**：若 `book_name` 为空，后端自动抓取文件名作为书名存储；从 Token 解析 `user_id`，并将生成的 `book_id`、`task_id` 与该 `user_id` 绑定存入数据库；生成任务并返回 ID。

**输出示例（JSON）**
```json
{
  "task_id": "t_001",
  "book_id": "b_123",
  "status": "pending"
}
```

---
### 处理进度查询
- **方法**：GET
- **路径**：`/api/v1/task/{task_id}`
- **主要输入参数**：`task_id`（路径参数）
- **业务逻辑**：从 Token 解析当前登录的 `user_id`，校验该 `task_id` 是否归属当前用户（非归属则返回 403/404）；返回对应 L0 存储、L1 章节解析等阶段的实时进度可视化。

**输出示例（JSON）**
```json
{
  "progress": 65,
  "stage": "L1_Parsing",
  "status": "running"
}
```

---
### 手动触发提取
- **方法**：POST
- **路径**：`/api/v1/extract`
- **主要输入参数**：
  - `book_id`: str
  - `roles`: list
  - `is_dynamic`: bool
- **业务逻辑**：从 Token 解析当前登录的 `user_id`，校验传入的 `book_id` 是否归属当前用户（非归属则拒绝执行）；根据总表勾选角色进行深度提取。`is_dynamic` 开启则识别形象改变节点。

**输出示例（JSON）**
```json
{
  "task_id": "t_002",
  "message": "Extraction started"
}
```

# 👤 3. 角色库与动态形象接口 (Character Persona)
**功能**：核心业务逻辑，处理「全局形象」与「动态形象」的切换。
> 注：需在 Header 中携带 `Authorization: Bearer <Token>`，后端自动过滤当前 `user_id` 下的数据。
### 已生成角色库
- **方法**：GET
- **路径**：`/api/v1/characters`
- **说明**：获取当前 `user_id` 下所有书籍中已完成提取的角色卡快照

**输出示例（JSON）**
```json
{
  "total": 42,
  "characters": [
    {
      "char_id": "c_01",
      "name": "石野",
      "from_book": "神游",
      "book_id": "b_123",
      "avatar_url": "/assets/chars/c_01.png"
    }
  ]
}
```

---
### 角色详情（全局）
- **方法**：GET
- **路径**：`/api/v1/characters/{char_id}`
- **说明**：获取该角色的 L2 级全局静态画像（基础属性、性格、核心设定）；自动校验该角色归属当前 `user_id`
	number代表编号，0代表全局静态画像，其他编号依据chapter_range进行依次编号
	chapter_range代表符合动态画像的章节范围（如果number为0，则chapter_range也为0即可）

**输出示例（JSON）**
```json
{
  "char_id": "c_01",
  "number": "0",
  "chapter_range": "0",
  "detail": "【姓名】【性别】【年龄】【种族】【能力】【社会关系】等这些详细的角色信息"
}
```
