# CharPick 数据库表结构汇总（**带精简注释版**）
所有表统一放在 **单个 PostgreSQL 库**，通过 **user_id 做数据隔离**，直接对接前端接口。

| 层级     | 对应业务表                                                                             | 这层存什么                      |
| ------ | --------------------------------------------------------------------------------- | -------------------------- |
| L0 原始层 | books、source_files                                                                | 书籍主入口、原始文件、导入元数据、OCR 来源    |
| L1 章节层 | chapters                                                                          | 切分后的章节文本、章节编号、章节范围、章节定位    |
| L2 提取层 | chapter_extractions                                                               | 每章抽出的角色、物品、剧情、地点、时间线 JSONB |
| L3 汇总层 | characters、character_versions、items、storyline_events、world_locations、media_assets | 面向前端和角色库的最终业务结果            |
| 通用支撑   | users、auth_sessions、extraction_tasks、task_logs                                    | 用户身份、登录态、任务过程、任务日志         |

## 一、用户与认证表
### 1. users（用户主表）
- id（PK）# 用户唯一ID，主键
- username # 用户名
- email # 邮箱
- password_hash # 密码哈希（不存明文）
- status # 账号状态：active/disabled
- created_at # 创建时间
- updated_at # 更新时间

### 2. auth_sessions（会话/Token 表）
- id（PK）# 会话唯一ID
- user_id（FK → users）# 所属用户（外键）
- access_token_jti # 访问令牌唯一标识
- refresh_token_hash # 刷新令牌哈希
- expires_at # 过期时间
- revoked_at # 撤销时间（NULL=有效）
- created_at # 登录时间
- updated_at # 最后更新时间

### 3. created_at / updated_at 自动维护
建议 `users` 和 `auth_sessions` 都由数据库自动维护时间字段：

- `created_at` 默认值为 `now()`，表示记录创建时间。
- `updated_at` 默认值为 `now()`，并在每次 `UPDATE` 时自动刷新。

可执行 SQL：

```sql
alter table public.users
alter column created_at set default now(),
alter column updated_at set default now();

alter table public.auth_sessions
alter column created_at set default now(),
alter column updated_at set default now();

create or replace function public.set_updated_at()
returns trigger
language plpgsql
as $$
begin
	new.updated_at = now();
	return new;
end;
$$;

drop trigger if exists trg_users_set_updated_at on public.users;
create trigger trg_users_set_updated_at
before update on public.users
for each row execute function public.set_updated_at();

drop trigger if exists trg_auth_sessions_set_updated_at on public.auth_sessions;
create trigger trg_auth_sessions_set_updated_at
before update on public.auth_sessions
for each row execute function public.set_updated_at();
```

前端配合规则：
- 插入 users / auth_sessions 时可以不传 `created_at`、`updated_at`。
- 如果前端已经传了时间，也不会影响，但建议以后都交给数据库维护。

### 4. users.status 约束修复方案
如果注册时报 `new row for relation "users" violates check constraint "users_status_check"`，说明数据库里的状态约束和前端写入值不一致。建议在 Supabase SQL Editor 执行下面这段，把状态值统一成 `active` / `disabled`：

```sql
alter table public.users
alter column status set default 'active';

update public.users
set status = 'active'
where status is null
	or status not in ('active', 'disabled');

alter table public.users
drop constraint if exists users_status_check;

alter table public.users
add constraint users_status_check
check (status in ('active', 'disabled'));

alter table public.users
alter column status set not null;
```

前端配合规则：
- 注册时不要手动写 `status`，让数据库默认值生效。
- 登录时如果读到 `disabled`，直接禁止登录。
- 如果历史数据里有中文状态值，先执行上面的 `update` 语句统一修正。

### 5. auth_sessions 必要约束
为了让登录态更稳定，建议为 `auth_sessions` 增加下面这些约束和索引：

- `access_token_jti` 唯一索引，避免重复会话标识。
- `user_id` 普通索引，便于按用户查询会话。
- `expires_at` 普通索引，便于清理过期会话。
- `revoked_at` 允许为空，撤销时再写入时间。

可执行 SQL：

```sql
create unique index if not exists auth_sessions_access_token_jti_key
on public.auth_sessions (access_token_jti);

create index if not exists auth_sessions_user_id_idx
on public.auth_sessions (user_id);

create index if not exists auth_sessions_expires_at_idx
on public.auth_sessions (expires_at);
```

### 6. 推荐的完整迁移顺序
如果你要一次性把当前数据库修正到位，建议按这个顺序执行：

1. 修复 `users.status` 默认值和检查约束。
2. 补 `users.username` 唯一约束。
3. 补 `auth_sessions.access_token_jti` 唯一索引。
4. 补 `auth_sessions.user_id` 和 `auth_sessions.expires_at` 索引。

建议补充的业务约束：

```sql
create unique index if not exists users_username_key
on public.users (username);
```

### 7. 前端需要同步的修改点
- 注册：只传 `username`、`password`、`email`，不要再写 `status`。
- 登录：根据 `username` 找到 `email` 后，再走 Supabase Auth 登录。
- 用户禁用态：前端统一识别 `active` / `disabled`，兼容旧数据里的中文值只作为兜底。
- 错误提示：把 `users_status_check`、`email rate limit exceeded`、`invalid email` 这类报错转成中文提示，避免直接暴露底层错误。

## 二、书籍与原始文件表
### 3. books（书籍/项目主表）
- id（PK）# 书籍唯一ID
- user_id（FK → users）# 所属用户（外键）
- title # 书名/项目名
- author # 原著作者
- source_type # 来源类型：epub/txt/pdf/image
- language # 语言：zh-CN/en
- status # 处理状态：待解析/解析中/已完成/失败
- cover_url # 封面图片地址
- created_at # 创建时间
- updated_at # 更新时间

### 4. source_files（源文件表 L0）
- id（PK）# 源文件ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- file_name # 原始文件名
- file_url # 文件存储地址
- file_hash # 文件哈希（防重复）
- mime_type # 文件类型
- raw_text_url # 原始文本地址
- ocr_text_url # OCR 文本地址（如果是图片文件输入，则这个是图片 OCR 识别出来的文本文件地址）
- parse_status # 解析状态

## 三、章节与提取表（L1/L2）
### 5. chapters（章节表 L1）（章节切分得到的结果）
- id（PK）# 章节ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- chapter_no # 章节序号
- chapter_title # 章节标题
- chapter_range # 章节范围（精准标记这一章在**原始文件中的起止位置**）
- content_text # 章节纯文本
- word_count # 字数统计
- source_file_id（FK → source_files）# 来源文件
### 6. chapter_extractions（章节提取表 L2）
- id（PK）# 提取记录ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- chapter_id（FK → chapters）# 所属章节
- extractor_type # 提取类型
- extraction_json（JSONB）# 章节提取结构化数据
- embedding_vector（pgvector）# 向量数据
- prompt_version # 提示词版本
- model_name # 使用模型
- confidence # 置信度

补充extractor_type解释

|常见 `extractor_type` 枚举值|对应提取任务|
|---|---|
|`character`|角色提取（提取本章出现的角色、人设）|
|`plot`|剧情提取（提取本章的剧情事件、时间线）|
|`item`|物品提取（提取本章出现的道具、装备）|
|`location`|地点提取（提取本章出现的场景、地点）|
|`summary`|章节摘要提取（生成本章的总结）|
|`full`|全量提取（一次性提取角色 + 剧情 + 物品等所有内容）|

## 四、汇总结果表（L3 前端核心）
### 7. characters（角色总表，每一行 = 一个角色）
- id (PK) # 角色唯一 ID
- user_id (FK → users) # 所属用户
- book_id (FK → books) # 所属书籍
- name # 角色名（展示用）
- alias # 别名
- role_type # 主角 / 配角 / 反派
- appearance_count INT # 出场次数
- first_appearance_chapter_id (FK → chapters) # 首次登场章节
- static_profile_json (JSONB) # 固定人设
- dynamic_profile_json (JSONB) # 动态关系 / 经历
- avatar_url # 头像
- status # 状态（未生成角色卡/静态全局角色卡/完整动态角色卡）
- created_at # 记录第一次生成角色卡的时间
- updated_at

### 8. character_versions（角色卡表）
- id（PK）# 版本ID
- user_id（FK → users）# 所属用户
- character_id（FK → characters）# 所属角色
- book_id（FK → books）# 所属书籍
- version_no # 版本号
- chapter_range # 适用章节范围
- profile_json（JSONB）# 该版本人设
- prompt_text # 生成用提示词
- is_default # 是否默认版本

### 9. items（物品表）
- id（PK）# 物品ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- name # 物品名称
- description # 描述
- owner_character_id（FK → characters）# 持有者
- first_appearance_chapter_id # 首次登场章节
- item_json（JSONB）# 物品详情

### 10. storyline_events（剧情时间线）
- id（PK）# 剧情事件ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- chapter_id（FK → chapters）# 所属章节
- event_title # 事件标题
- event_summary # 事件摘要
- participants_json（JSONB）# 参与角色列表
- location_id # 发生地点
- event_time # 事件时间

### 11. world_locations（世界观/地点表）
- id（PK）# 地点ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- name # 地点名称
- description # 描述
- related_chapter_ids（JSONB）# 关联章节
- location_type # 地点类型：城市/建筑/场景

## 五、媒体资源表
### 12. media_assets（图片/封面/角色立绘）
- id（PK）# 媒体资源ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- asset_type # 资源类型：封面/头像/表情包
- object_url # 原图地址
- thumbnail_url # 缩略图地址
- tags_json（JSONB）# 标签
- embedding_vector（pgvector）# 图片向量
- source_character_id（FK → characters）# 关联角色

## 六、任务与日志表
### 13. extraction_tasks（异步任务表）
- id（PK）# 任务ID
- user_id（FK → users）# 所属用户
- book_id（FK → books）# 所属书籍
- task_type # 任务类型：上传/解析/提取/汇总
- status # 任务状态
- progress # 进度百分比
- error_message # 错误信息
- created_at # 创建时间
- updated_at # 更新时间

### 14. task_logs（任务日志）
- id（PK）# 日志ID
- task_id（FK → extraction_tasks）# 所属任务
- user_id（FK → users）# 所属用户
- level # 日志级别
- message # 日志信息
- detail_json（JSONB）# 详细数据
- created_at # 记录时间
