# NovelWeaver 前端技术路径

本文档基于当前 `frontend_vue` 工程整理，目标是把前端从“页面展示层”明确成一条可持续迭代的技术路径，覆盖登录、上传、任务流、角色库、聊天和配置管理。

## 1. 前端定位

前端的职责不是单纯展示数据，而是承接小说内容从导入、解析、提取、汇总到交互使用的完整工作流。

当前前端已经具备的能力包括：

- 登录与会话维护
- 文件上传到 OSS
- 任务状态查看与提取触发
- 角色卡与摘要结果展示
- 聊天交互与全局配置管理

## 2. 当前技术栈

前端工程采用 Vue 3 体系，整体技术选型如下：

- 核心框架：Vue 3
- 构建工具：Vite
- 路由：Vue Router 4
- 状态管理：Pinia
- UI 组件：Element Plus
- 请求层：Axios
- 存储与后端能力：阿里云 OSS、Supabase

## 3. 目录与职责划分

当前 `src` 目录已经形成比较清晰的分层：

- `src/pages/`：页面级视图，负责业务流程组织
- `src/components/`：通用布局与可复用组件
- `src/stores/`：全局状态与本地持久化状态
- `src/api/`：后端 API 封装与业务请求编排
- `src/lib/`：第三方服务初始化与通用能力封装

其中比较关键的模块有：

- `src/router.js`：统一路由入口与权限控制
- `src/App.vue`：页面壳层与布局切换
- `src/stores/auth.js`：登录态、用户信息、会话恢复
- `src/stores/settings.js`：前端配置、模型参数、OSS/接口参数
- `src/stores/appState.js`：当前书籍、当前上下文等业务状态
- `src/api/client.js`：统一导出 API 对象
- `src/lib/oss.js`：OSS 客户端与配置校验
- `src/lib/supabase.js`：Supabase 客户端与会话能力

## 4. 前端架构路径

### 4.1 入口层

应用从 `src/main.js` 启动，完成 Vue、Pinia、路由和 Element Plus 的挂载。这里的目标是把全局依赖一次性注入，确保页面层只关注业务。

### 4.2 路由层

路由由 `src/router.js` 统一管理，当前路由结构已经清楚地分成了：

- `login`：独立登录页
- `upload`：文件上传与处理页
- `dashboard`：任务与摘要看板
- `characters`：角色卡管理页
- `chat`：对话页，支持保活
- `settings`：全局配置页

路由守卫的核心作用是：

- 未登录时拦截并跳转到登录页
- 已登录后自动恢复用户信息
- 维持页面访问前的一致会话状态

### 4.3 页面层

页面层的设计不是简单列表页，而是围绕“小说处理链路”拆分：

- 登录页负责建立会话
- 上传页负责把文件送入 OSS 并触发处理任务
- 看板页负责查看整体处理进度、摘要和任务日志
- 角色页负责展示提取后的角色卡和归档内容
- 聊天页负责基于当前书籍上下文进行交互
- 设置页负责维护模型、接口和上传相关参数

### 4.4 状态层

Pinia 是前端状态中枢，当前分工比较明确：

- `auth`：保存 token、用户信息、登录状态
- `appState`：保存当前书籍、当前上下文和页面间共享状态
- `settings`：保存模型参数、接口地址、OSS 配置和个性化设置

这套设计适合当前项目的原因是：

- 登录态需要在刷新后恢复
- 当前书籍需要在不同页面间共享
- 设置项需要持久化到本地

### 4.5 API 层

`src/api/` 按业务域拆分请求逻辑，避免页面直接拼接请求细节。当前已经拆分出：

- `authApi`
- `bookApi`
- `taskApi`
- `summaryApi`
- `cardApi`
- `chatApi`

对应的基础工具放在 `src/api/core.js`，负责处理：

- Base URL 归一化
- OpenAI 兼容地址判断
- 代理地址拼装
- 请求头构建

### 4.6 OSS 与外部能力

文件类业务的关键在于先上传到 OSS，再由后端或任务系统接手处理，所以前端需要在本地把 OSS 能力封装清楚：

- `src/lib/oss.js` 负责配置读取、校验和客户端初始化
- `src/api/oss/` 负责具体上传、删除、回滚和资源编排
- `src/lib/supabase.js` 负责 Supabase 连接与匿名会话能力

## 5. 前端技术路径建议

结合当前工程状态，前端后续可以按下面的路径迭代：

### 第一阶段：稳定基础链路

- 固化登录、会话恢复和权限控制
- 完善上传 -> OSS -> 任务触发 -> 状态刷新链路
- 统一请求错误处理和消息提示

### 第二阶段：增强业务可视化

- 将任务、摘要、角色卡、章节提取结果做成更强的聚合视图
- 对当前书籍上下文提供更明显的切换入口
- 让看板页承担“全局调度中心”的角色

### 第三阶段：强化创作交互

- 让聊天页基于当前书籍、角色卡和摘要结果进行上下文拼装
- 让角色页支持更多字段展示，例如外貌、性格、经典台词、关系网
- 让设置页支持模型参数和提示词模板的快速切换

### 第四阶段：面向多模态扩展

- 预留图片、表情包、角色立绘等资源展示区
- 预留上传后续多模态任务的入口
- 为后续文本生成图片、角色图鉴和音色联动保留状态字段和页面结构

## 6. 推荐实施顺序

如果要继续在当前前端上扩展，建议按以下优先级推进：

1. 先稳定认证与路由守卫，保证所有页面都能可靠进入。
2. 再统一上传、任务和摘要的状态流转，减少页面间重复请求。
3. 然后增强角色页和聊天页的上下文联动。
4. 最后再补多模态展示和更强的创作辅助能力。

## 7. 与后端协同的接口边界

前端建议只负责三类事情：

- 采集用户输入
- 调用后端或存储服务
- 展示和维护会话状态

不建议把下面这些逻辑写死在页面里：

- 具体解析算法
- 向量检索实现细节
- 角色提取 Prompt 的业务规则
- 长文本切片和模型调度策略

这些能力应由后端或专门的处理服务负责，前端只保留入口、配置和结果呈现。
